# 系统边界与结构分析

> "现代科学技术不是单单研究一个个事物、一个个现象，而是研究事物、现象的变化发展过程，研究事物相互之间的关系。"
> —— 钱学森《现代科学技术的特点和体系结构》

## 核心原则

**结构决定功能。一个系统的结构分析如果不清晰，后续所有的建模和执行都将建立在流沙之上。子系统划分不合理、接口定义不明确，是复杂工程失败的首要原因。**

---

## 不适用场景

以下情况**不需要**调用本 skill：
- 单体系统，组分数量 ≤ 3
- 结构已经明确且无争议
- 当前任务不需要深入到结构层面
- 用户已经提供了详细的结构文档

---

## 方法论流程

### 第一步：层次分解

> 钱学森将现代科学技术分为四个层次：工程技术 → 技术科学 → 基础科学 → 马克思主义哲学

**对系统进行层次化的树状分解：**

```
[系统名称]
├── 子系统A
│   ├── 组件A1
│   │   ├── 单元A1a
│   │   └── 单元A1b
│   └── 组件A2
├── 子系统B
│   ├── 组件B1
│   └── 组件B2
└── 子系统C
    └── 组件C1
```

**关键规则：**
- **内聚原则**：同一子系统内的组件功能紧密相关
- **耦合原则**：子系统之间的接口尽量少、尽量简单
- **完整性**：所有系统内组分都要被纳入，不遗漏

### 第二步：定义每个组分的规格

> 钱学森用一个工厂、一个企业、一个机构、一个单位的系统工程来比喻"小范围"、"小系统"，强调每个组分都要有明确的职责

**使用统一规格表：**

```
[组分名称]
├─ 功能：[一句话描述]
├─ 输入：
│   ├─ 输入1 (类型，来源，频率)
│   └─ 输入2 (类型，来源，频率)
├─ 输出：
│   ├─ 输出1 (类型，去向，频率)
│   └─ 输出2 (类型，去向，频率)
├─ 内部状态：[需要维护的状态变量]
├─ 配置参数：[可外部调整的参数]
├─ 性能指标：[关键KPIs]
└─ 约束条件：[不可违反的约束]
```

### 第三步：接口契约定义

> 钱学森强调计量工作和标准化："如果你的东西跟别人对不上，生产就搞不下去，那就必须要有计量工作。"

**为每个跨组分接口定义明确的契约：**

```
接口：[接口名]
├─ 提供方：[组分名]
├─ 消费方：[组分名]
├─ 数据格式：[schema]
├─ 传输方式：[同步/异步/批量]
├─ 频率：[每秒/每步/按需]
├─ SLA：[延迟上限、吞吐量要求、可用性]
├─ 错误处理：[重试策略、降级方案、fallback]
└─ 版本：[接口版本号]
```

### 第四步：系统边界检查

**对系统边界进行六维检查：**

| 维度 | 检查问题 | 状态 |
|------|---------|------|
| 物理边界 | 所有物理实体是否被纳入？硬件/软件/人/环境？ | ✓/✗ |
| 逻辑边界 | 所有逻辑功能是否被覆盖？数据流是否完整？ | ✓/✗ |
| 时间边界 | 系统的生命周期阶段是否明确？（设计/开发/运行/退役） | ✓/✗ |
| 组织边界 | 哪些团队/人员负责哪些组分？权责是否清晰？ | ✓/✗ |
| 知识边界 | 系统的哪些部分已知（有模型支持）？哪些未知？ | ✓/✗ |
| 场景边界 | 系统的使用场景是什么？（谁用、在哪用、怎么用、多久用一次）场景对系统施加了哪些不可转移的约束？（如复用要求、长时间滑行、自生增压） | ✓/✗ |

## 常见结构反模式

| 反模式 | 症状 | 书中的警示 | 修正方法 |
|--------|------|----------|---------|
| 过度耦合 | 改一处动全身 | "各组成部分间的相互关系暂时撇开"会导致错误 | 解耦，定义清晰接口 |
| 层次混乱 | 高层管细节、低层做决策 | 必须"能动地推动"各层各司其职 | 恢复层次关系 |
| 遗漏组分 | 重要功能没有归属 | 系统定义不完整 | 重新遍历边界 |
| 接口不标准 | 组分间对不上 | "没有计量工作的现代化，要实现四个现代化是不可能的" | 标准化接口 |
| 边界不清 | 无法判断谁负责什么 | 不能"把自然辩证法的研究范围扩大到远远超出恩格斯的原意" | 明确定义边界 |

## 与其他 skill 的关系

### 系统工程链内

- **上游**：建立在 `systems-thinking-framework` 的系统思维报告基础上
- **下游**：产出结构分析报告供 `quantitative-modeling-workflow` 使用
- **平行**：与 `investigation-first`（工程扩展模式）平行调用，结构分析 + 领域调研同时进行

### 与 qiushi 思想方法层深度耦合

本 skill 与 qiushi 的 `overall-planning` 构成**结构分析-全局平衡**的联合决策链：

| 本 skill 的步骤 | 联动 qiushi skill | 联动方式 |
|---------------|------------------|---------|
| 第一步（层次分解） | `contradiction-analysis` | 子系统划分的合理性→分辨子系统间的矛盾性质 |
| 第二步（组分规格） | `overall-planning` | 各组分的功能权重→是否为某对辩证关系中的主要方面 |
| 第三步（接口契约） | `mass-line` | 接口设计的双向性→"从群众中来，到群众中去"的工程体现 |
| 第四步（六维边界检查） | `overall-planning` | 检查是否存在片面性倾向（某维度过度、某维度缺失） |

**联合调用协议：**
1. `overall-planning` 先输出辩证关系全景图
2. 本 skill 基于该平衡方案进行结构分析
3. 六维边界检查表中增加"辩证平衡检查"维度：
   - 物理边界 ⇄ 速度 vs 质量的偏向
   - 组织边界 ⇄ 集中统一 vs 分散自主的取舍
   - 知识边界 ⇄ 探索创新 vs 稳定可靠的权衡
4. 结构反模式中发现"过度耦合"或"边界不清"时，返回 `overall-planning` 检查是否是辩证关系失衡的表现
