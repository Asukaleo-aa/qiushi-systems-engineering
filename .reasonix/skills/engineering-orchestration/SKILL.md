# 工程编排与协调

> "系统工程是要解决具体实际问题的，是讲实干的，是要改造客观世界的，而不是只研究研究学问。"
> —— 钱学森《什么叫系统工程》

> "聂荣臻同志那时也是国务院副总理，国家科学技术委员会主任，国防科学技术委员会主任，统筹兼顾，全面调度，充分发挥了科学技术人员的聪明才干，研制工作取得了迅速的进展。"
> —— 钱学森《聂荣臻同志开创了中国大规模科学技术研制工作的现代化组织管理》

## 核心原则

**再好的设计，如果执行不力也是空谈。工程编排是将系统思维转化为可操作任务的能力——借鉴聂荣臻组织"两弹一星"的成功经验，通过统筹兼顾、全面调度，将复杂系统的实施转化为有序推进的任务序列。**

---

## 不适用场景

以下情况**不需要**调用本 skill：
- 任务极其简单（少于3个步骤且无依赖）
- 系统设计尚未完成或未通过验证
- 纯探索/研究性质的工作（没有明确的交付目标）
- 一次性操作且不需要重复执行

## 方法论流程

### 第一步：工作分解结构

> 钱学森借鉴美国北极星导弹研制的PERT（计划评审技术）方法

**从系统结构（参考 system-boundary-structuring 的产出）映射到工作任务：**

```
系统分解结构（SBS） → 工作分解结构（WBS）

系统
├── 子系统A ──────────────
│   ├── 组件A1
│   │   ├── 任务A1.1：[描述]
│   │   │   ├── 子任务A1.1.1
│   │   │   └── 子任务A1.1.2
│   │   └── 任务A1.2
│   └── 组件A2 ...
├── 子系统B ...
└── 集成与验证 ...
    ├── 集成测试
    └── 系统验收
```

**每个任务必须包含：**
- 任务描述
- 负责方（人或agent）
- 输入（前置产出物）
- 产出物
- 预计工作量
- 验收标准

### 第二步：依赖关系分析

**使用邻接矩阵或前驱图分析任务依赖：**

```
前驱关系图：
A1.1 → A1.2 → A_Integration
B1.1 ↗    B1.2 ↗

关键分析：
- 哪些任务是串行的（必须等待前置完成）？
- 哪些任务是并行的（可以同时进行）？
- 哪些任务是双向依赖（需要协同迭代）？
```

**依赖类型标注：**
- `FS`（Finish-to-Start）：A完成 → B开始（最常见）
- `SS`（Start-to-Start）：A开始 → B可开始
- `FF`（Finish-to-Finish）：B必须与A同时完成
- `SF`（Start-to-Finish）：A开始 → B才能完成（最罕见）

### 第三步：关键路径识别

> PERT方法的核心——识别决定总工期的关键路径

```
对每条路径计算总时间：
路径1 (A1→A2→A3→集成) = 3 + 5 + 2 + 4 = 14天 ← 关键路径
路径2 (B1→B2→集成)     = 2 + 3 + 4 = 9天
路径3 (C1→C2)           = 1 + 2 = 3天

关键路径上的任何延迟 → 整体项目延迟
非关键路径上的延迟 → 如果不超过缓冲时间，不影响整体
```

### 第四步：执行顺序规划

**参考霍尔三维结构的"时间维"：**

```
阶段划分：
┌──────────────────────────────────────────┐
│ 规划 → 设计 → 开发 → 集成 → 验证 → 交付 │
└──────────────────────────────────────────┘

每个阶段定义：
- 准入条件（进入该阶段前必须完成什么）
- 退出条件（完成该阶段的验收标准）
- 检查点（Gate Review）
```

**质量门（Quality Gates）：**
- **Gate 1**（规划完成后）：方向正确吗？资源到位吗？
- **Gate 2**（设计完成后）：设计充分吗？有没有遗漏？
- **Gate 3**（开发完成后）：组件通过单元测试了吗？
- **Gate 4**（集成完成后）：系统通过集成测试了吗？
- **Gate 5**（验证完成后）：满足系统目标吗？可以交付吗？

### 第五步：风险识别与缓解

> "在一切失误中，决策失误是最大的失误；在一切成功中，决策的成功是最大的成功。" —— 于景元《系统工程在决策中的应用》

```
风险矩阵：
| 风险 | 概率 | 影响 | 等级 | 缓解措施 | 触发条件 | 应急预案 |
|------|------|------|------|---------|---------|---------|
| R1   | 高/中/低 | 高/中/低 | 红/黄/绿 | ... | ... | ... |
```

**尤其注意：**
- 关键路径上的风险（任何延迟直接拖累总工期）
- 耦合点的风险（多个任务在一点交汇）
- 知识缺口导致的风险（未知领域的不确定性）

### 第六步：资源调度

> "统筹兼顾，全面调度" —— 聂荣臻组织两弹一星的核心经验

**资源负载分析：**
- 哪些agent/人同时承担多个任务？
- 有没有资源争抢（同一资源同时需要做两件事）？
- 关键资源是否集中在关键路径上？

## 强制输出

```
## 工程编排计划

### 1. 工作分解结构（WBS）
[树状分解 + 任务规格表]

### 2. 依赖关系
[前驱图 + 依赖类型标注]

### 3. 关键路径
[路径分析 + 总工期估算 + 缓冲时间]

### 4. 阶段规划
[阶段划分 + 准入/退出条件 + 质量门]

### 5. 风险矩阵
[表格：风险 | 概率 | 影响 | 等级 | 缓解 | 触发 | 应急]

### 6. 执行命令
按优先级排序的任务列表（含负责方和截止时间）

### 7. 监控指标
[需要追踪的KPIs和里程碑]
```

---

## 编排铁律

1. **关键路径优先** —— 关键路径上的任务享有最高优先级
2. **质量门不可跳过** —— 每个Gate的验收标准必须满足
3. **依赖不可颠倒** —— 后续任务不能在前置任务完成前开始
4. **缓冲必须预留** —— 不要在计划中把时间排到100%
5. **可见性必须保持** —— 所有参与方必须随时了解进度和阻塞

## 与其他 skill 的关系

### 系统工程链内

- **上游**：接收 `systems-thinking-framework` 的系统理解、`simulation-validation-cycle` 的验证结论、`meta-synthesis-engine` 的综合判断
- **执行**：产出的编排计划直接驱动实际的开发执行
- **监控**：执行过程中的偏差触发 `feedback-and-revision-loop`

### 与 qiushi 思想方法层深度耦合

本 skill 与 qiushi 的 `concentrate-forces` 构成**战略-战术双层执行体系**：

```
engineering-orchestration （战术层：全局编排）
    ↓ 产出：WBS + 关键路径 + 质量门
concentrate-forces （战略层：在关键路径上选主攻方向）
    ↓ 宣告：当前战役目标 + 完成信号
[执行...]
    ↓ 每过一道质量门
concentrate-forces （重新评估：是否转向下一个目标？）
    ↓
criticism-self-criticism + feedback-and-revision-loop （阶段复盘）
```

**具体耦合点：**

| 本 skill 的步骤 | 联动 qiushi skill | 联动方式 |
|---------------|------------------|---------|
| 第一步（WBS） | `contradiction-analysis` | WBS 任务树中识别主要矛盾——哪个子系统是瓶颈？ |
| 第三步（关键路径） | `concentrate-forces` | 在关键路径上确定主攻方向（"集中优势兵力"） |
| 第四步（质量门） | `criticism-self-criticism` | 每个 Gate 检查触发工作质量审视 |
| 第五步（风险矩阵） | `investigation-first` | 知识缺口导致的风险→补充调研 |
| 第六步（资源调度） | `overall-planning` | 多 agent 资源分配→统筹兼顾各方 |
| 全文（执行纪律） | `practice-cognition` | "实践是检验真理的唯一标准"→编排计划在实践中修正 |
