# 反馈与修正循环

> "人们在实践中对于客观现实的认识是永远不会完结的，但有阶段性，表现为认识的能动的飞跃。"
> —— 钱学森《关于系统工程与经济管理体制》

> "一般说来，在任何一门现代精密科学中，对数量关系的研究是科学概念形成的基础，在社会科学中也不应例外。"
> —— 宋健《社会科学研究的定量方法》

## 核心原则

**系统工程是一个动态的、螺旋上升的认知过程，不是线性的执行链条。没有反馈的系统是盲目的；只有建立有效的反馈和修正机制，系统才能在运行中自我完善。**

本 skill 是全部系统工程流程的最后一个环节——它将实践结果反向注入到理论认识中，形成钱学森所说的"认识 → 实践 → 再认识"的螺旋上升。

---

## 不适用场景

以下情况**不需要**调用本 skill：
- 工作尚未到达任何检查点（没有需要审查的产出）
- 一次性任务且没有学习价值
- 执行刚刚开始，尚无偏差需要反馈
- 系统已经稳定运行且没有变化

## 方法论流程

### 第一步：触发条件识别

**本 skill 在以下时机被激活：**

1. **阶段完成触发**：
   - 任何主要阶段（规划/设计/建模/仿真/编排）完成后
   - 任何质量门检查后
   - 任何重大决策实施后

2. **偏差触发**：
   - 实际结果与预期偏离超过阈值
   - 关键路径上的任务延迟超过缓冲时间
   - 发现了之前未识别的风险或涌现行为

3. **定期触发**：
   - 按预定周期（如每日/每周）进行系统健康检查

### 第二步：偏差量化

> "如果要实事求是，就得有真实的数据。" 反馈判断不能仅凭直觉，必须定量

**对每个偏差进行量化分析：**

```
偏差分析矩阵：

偏差ID：[编号]
发现时机：[在哪个阶段/哪个检查点发现的]
原始预期：[预期值是什么]
实际结果：[实际值是什么]
偏差量：[数学化的差异]
偏差方向：[正向偏差（好于预期）/ 负向偏差（差于预期）]
偏差程度：[轻微（<5%）/ 中等（5-20%）/ 严重（>20%）]
首次发现：[时间]
持续趋势：[恶化中/稳定/改善中]
```

### 第三步：根因追溯

> "仅仅停留在分析的方法上，是不够的，应该把客观世界看作一个统一的、在相互关联中变化的集合体。" —— 钱学森《略谈系统科学》

**五层根因分析法：**

```
表面现象：系统运行慢了20%
↓ Why 1: 为什么慢了？
→ 因为某个组件的处理时间增加了50%
↓ Why 2: 为什么处理时间增加了？
→ 因为输入数据量增大了3倍
↓ Why 3: 为什么数据量增大导致非线性增长？
→ 因为算法复杂度是O(n²)，在建模时假设了O(n)
↓ Why 4: 为什么假设了O(n)？
→ 因为在仿真阶段测试的最大数据量只有实际的1/10
↓ Why 5 (根本原因): 为什么仿真数据量只有实际的1/10？
→ 仿真场景设计中遗漏了大规模数据场景

修复点：
1. 立即：实现O(n log n)算法
2. 制度：在仿真场景标准中增加"10倍正常负载"的压力测试
3. 文化：在简化假设评审中增加"假设失效条件"检查
```

**根因可能来源的层级（对应钱学森四层次 + 场景层）：**
- 哲学层：指导思想或基本假设有误
- 理论层：模型原理有缺陷
- 场景层：错误源于在错误的系统层次上定义了问题——把产品当物理系统优化、在物理层求解了一个应该在产品层定义的目标、忽略了使用场景施加的不可转移约束
- 方法层：技术方法选择不当
- 执行层：实施过程中出现了错误

### 第四步：修正方案制定

**对每个根因，制定三级修正方案：**

```
短期修正（止血）：
- 做什么：[临时方案，立即缓解症状]
- 时间：[几小时内/几天内]
- 风险：[临时方案的副作用]

中期修正（治本）：
- 做什么：[解决根本原因]
- 时间：[几天到几周]
- 影响范围：[会影响哪些已完成的组件]

长期修正（免疫）：
- 做什么：[预防同类问题再次发生]
- 是什么制度性改进？
- 应该加到哪个流程/检查清单中？
```

### 第五步：认知更新

> "人的认识不是沿着直线进行的，而是无限地近似于一串圆圈、近似于螺旋的曲线。" —— 这是一个认识的能动的飞跃

**将修正的经验更新到系统认知中：**

```
认知更新记录：
- 更新内容：[新发现了什么规律/关系]
- 修正了之前的哪个认知：[原来是...，现在是...]
- 应更新到哪个文档：[系统思维报告 / 结构分析 / 调研报告 / 模型 / 编排计划]
- 对哪些假设产生了影响：[列出受影响的假设]
```

### 第六步：决定后续行动

```
决策分支：
                          根因层是？
                          /   |    \
                    哲学/理论层 场景层 方法/执行层
                    /            |           \
            → 返回系统思维框架   → 返回系统思维    ├─ 方法有误 → 返回定量建模
              重新审视基本假设   框架第零步，      ├─ 执行有误 → 返回工程编排
                                重新锚定系统层次   └─ 认知不足 → 返回问题域调研
                                （把产品当物理系统
                                优化了？目标定义
                                在错误层次上？）
```

## 强制输出

```
## 反馈与修正报告

### 1. 触发事件
[什么触发了这次反馈审查]

### 2. 偏差分析
[偏差量化表和根因追溯链]

### 3. 根因总结
[根本原因 + 发生在哪个层次]

### 4. 修正方案
[短/中/长期修正方案]

### 5. 认知更新
[更新了哪些认识 + 影响了哪些文档]

### 6. 回归验证
[修正后需要重新验证的内容]

### 7. 经验教训
[值得记录供未来参考的经验]

### 8. 下一步行动
[决定返回哪个上游 skill + 具体行动项]
```

---

## 反馈循环铁律

1. **偏差不隐藏** —— 所有偏差都必须报告，正面偏差（超预期）也需要分析
2. **根因不表面** —— 不满足于直接原因，必须追溯至少5层
3. **修正有层次** —— 止血+治本+免疫，三层缺一不可
4. **认知必更新** —— 每次修正必须写回系统文档
5. **闭环后再前进** —— 修正完成、回归验证通过后，才继续推进

## 与其他 skill 的关系

**本 skill 是整个系统工程方法论体系的"缝合线"——它连接任意上游和下游 skill：**

### 系统工程链内

- 从任何 skill 的产出偏差都可以触发
- 根据根因层级，引导返回到对应的上游 skill 进行修正

### 与 qiushi 思想方法层深度耦合

本 skill 与 qiushi 的 `criticism-self-criticism` 构成**人-系统双重反馈闭环**：

| 维度 | `feedback-and-revision-loop`（本 skill，系统维度） | `criticism-self-criticism`（人维度） |
|------|--------------------------------------------------|------------------------------------|
| 审视对象 | **系统行为偏差**和模型误差 | Agent 的**工作方法和决策质量** |
| 触发信号 | 实际结果 vs 预期偏离超过阈值 | 阶段完成 / 收到反馈 / 反复出错 |
| 分析方法 | 五层根因分析（Why1→Why5） | 自我批评检查清单 → 具体行为归因 |
| 修正输出 | 短/中/长期三级修正 + 认知更新 | "下次重点关注"清单 |
| 认识论基础 | 钱学森"认识→实践→再认识" | 毛泽东"治病救人" |

**联合闭环机制：**

```
[偏差触发]
    ↓
feedback-and-revision-loop （系统视角：根因在哪？）
    ├─ 哲学/理论层 → 返回 systems-thinking-framework 重审假设
    ├─ 方法/执行层 → 返回对应 skill 修正
    └─ 同时触发 ↓
criticism-self-criticism （人的视角：我错过了什么？）
    ├─ 是否跳过了本该做的调查/验证步骤？
    ├─ 是否分散了注意力？
    └─ 是否先入为主忽略了反面证据？
```

**与 `practice-cognition` 的协同：** 本 skill 的"偏差→修正→再检验"流程，正是 `practice-cognition` 中"实践→认识→再实践→再认识"螺旋的工程技术实现。两个 skill 共同构成从认识到实践、从实践返回认识的双向通道。
