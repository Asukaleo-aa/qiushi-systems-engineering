# 递阶分解协调

> "某一种控制作用对一个局部是好的，而从全局来看可能是不好的甚至是不允许的。"
> —— 钱学森、宋健《工程控制论》第21章

## 核心原则

**将整体任务分解为自治子系统（局部控制器），增设协调层只处理耦合约束，在递阶结构中实现局部自治+全局优化。**

- **子系统不互相窥探内部**：每个子系统有自己的目标函数和决策空间，局部控制器独立运行
- **协调层不干预子系统内部**：只传递约束松弛量/影子价格/耦合变量，子系统自行吸纳

> 详细原著依据见《工程控制论》第21.2—21.6节。

## 不适用场景

以下情况**不需要**调用本 skill：
- 只有1-2个子任务，无耦合——直接执行即可
- 子系统间完全独立，不存在共享资源或公共约束
- 任务已经被合理拆分且协调机制运转正常
- 需要强实时性的紧耦合系统——协调层的迭代通信可能太慢

## 何时使用

- 将大规模任务分解为多个子agent，但子agent的输出互相影响
- 多个子系统共享有限资源（算力、时间、上下文窗口）
- 局部优化方案汇总后出现冲突
- 需要"分而治之"但缺乏整合机制
- 在 `dispatching-parallel-agents` 后子agent互不通讯导致全局不优

## 方法流程

### 第一步：辨识耦合变量

写下所有子系统输入输出的交叉依赖：

```
子系统A输出 → 子系统B输入: [具体变量]
子系统B输出 → 子系统A输入: [具体变量]
子系统A、B共享资源R: [竞争关系]
```

**关键规则**：只列出跨子系统的外部依赖，子系统内部的变量不纳入。

### 第二步：设立子系统局部目标

每个子系统获得独立的目标函数和约束集。**只包含该子系统能直接控制的变量。**

如果子系统i的原目标为 J_i(x_i, x_j)，将其改写为：
- 局部目标：J_i(x_i; z_{j→i})，其中 z_{j→i} 是其他子系统传递来的耦合变量当前值
- 局部约束：x_i ∈ X_i

### 第三步：设计协调策略

根据耦合类型选择协调机制：

| 耦合类型 | 协调策略 | 协调层传递 |
|---------|---------|-----------|
| 资源竞争 | 影子价格/拉格朗日乘子 | 资源单价 λ_k |
| 输入输出耦合 | 目标值交互 | 期望输出 z̄_{i→j} |
| 公共约束 | 约束松弛 | 容许违反量 δ |
| 时序依赖 | 里程碑同步 | 完成信号 |

**协调迭代模式**：
```
1. 协调层广播初始参数
2. 子系统独立求解，返回对本子系统不可行/非最优的反馈
3. 协调层汇总反馈，更新参数后重新广播
4. 重复至收敛或达到迭代次数上限
```

### 第四步：收敛判断

设定终止条件（满足其一即停）：
- 各子系统反馈不再变化（稳定）
- 全局指标改善低于阈值
- 达到预设迭代轮次上限
- 检测到振荡——切换到降级策略（取最近可行解）

> 对不可收敛情况，回退到 `contradiction-analysis` 重新判定主要矛盾：是否有子系统目标根本上冲突？是否需要放宽某个约束？

## 与其他skill的关系

**认知位置**：求解环。前置依赖「建模」环的 `system-boundary-structuring` 产出。

**同环工具**：
- 子系统间的最优协调本质上是一个约束优化问题 → `constrained-optimization`
- 协调不收敛时可能需要渐进逼近 → `perturbation-progressive-method`
- 换一个表述可能简化协调结构 → `multi-representation-equivalence-transform`

**回连上游**：
- 协调策略需要适配环境变化 → `adaptive-robust-strategy`
- 协调的目标是在整体上统筹兼顾各方 → `overall-planning`
- 当协调不收敛时，回到 `contradiction-analysis` 确认主要矛盾

## 常见错误

| 错误 | 正确做法 |
|------|---------|
| 协调层直接修改子系统的内部状态 | 只传递边界参数，子系统自行消化 |
| 在子系统内部处理跨系统耦合 | 抬头一层，耦合问题交给协调层 |
| 一次性分解后不加迭代 | 需要多次协调迭代才能收敛 |
| 用单目标优化处理多目标问题 | 对不可比较的指标用 `overall-planning` 做权衡 |
