# 检测与修正偏差

## 描述

系统工程是动态螺旋上升的认知过程，不是线性执行链条。本技能在阶段完成或偏差出现时激活，对偏差进行量化分析、根因追溯和三级修正（止血/治本/免疫），并将经验写回系统认知，形成闭环反馈。

## 使用场景

### 何时使用

- 任何主要阶段（规划/设计/建模/仿真/编排）完成后
- 实际结果与预期偏离超过阈值
- 关键路径任务延迟超过缓冲时间
- 发现之前未识别的风险或涌现行为
- 按预定周期进行系统健康检查

### 何时不使用

- 工作尚未到达任何检查点，没有需要审查的产出
- 一次性任务且没有学习价值
- 执行刚刚开始，尚无偏差需要反馈
- 系统已稳定运行且无变化

## 输入输出

### Input

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| trigger_event | string | 是 | 触发本次反馈审查的事件描述 |
| expected_result | any | 是 | 原始预期值或状态 |
| actual_result | any | 是 | 实际观测到的值或状态 |
| deviation_threshold | number | 否 | 偏差阈值百分比，默认 5% |
| current_stage | string | 是 | 当前所处阶段（规划/设计/建模/仿真/编排） |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| deviation_analysis | object | 偏差量化表（偏差量、方向、程度、趋势） |
| root_cause | object | 根因追溯链 + 根因层级（哲学/理论/场景/方法/执行） |
| corrections | object | 三级修正方案（短期止血/中期治本/长期免疫） |
| cognition_updates | array | 认知更新记录（修正了什么、应更新到哪个文档） |
| next_action | string | 决定返回哪个上游 skill + 具体行动项 |

## 指令

1. **识别触发条件**：判定触发类型——阶段完成触发、偏差触发或定期触发。
2. **量化偏差**：对每个偏差填写偏差分析矩阵——原始预期、实际结果、偏差量、偏差方向（正向/负向）、偏差程度（轻微<5%/中等5-20%/严重>20%）、持续趋势（恶化/稳定/改善）。
3. **追溯根因**：对每个偏差执行五层 Why 分析，直至定位根本原因；将根因归入五个层级之一：哲学层（指导思想有误）、理论层（模型原理缺陷）、场景层（在错误系统层次定义问题）、方法层（技术选择不当）、执行层（实施出错）。
4. **制定三级修正**：短期修正——临时方案，几小时至几天内止血，标注副作用风险；中期修正——解决根本原因，几天至几周，标注影响范围；长期修正——制度性改进，预防同类问题再发，明确应加入哪个流程或检查清单。
5. **更新认知**：将修正经验写入认知更新记录——新发现的规律、修正了之前的哪个认知、应更新到哪个文档、对哪些假设产生影响。
6. **决定后续行动**：根据根因层级选择回归路径——哲学/理论层返回系统思维框架重审假设，场景层返回系统思维框架第零步重新锚定系统层次，方法层返回定量建模，执行层返回工程编排，认知不足返回问题域调研。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 偏差无法量化（缺少基线或预期值） | 先调用 investigation-first 收集基线数据，再重新执行偏差量化 |
| 五层 Why 仍无法定位根因 | 标记为"未知根因"，启动 investigation-first 实地调查，同时执行短期止血方案 |
| 修正方案与现有系统冲突 | 优先执行短期止血，中期修正提交 overall-planning 进行多目标权衡后再实施 |
| 正面偏差（超预期）被忽略 | 强制要求分析所有偏差，包括正向偏差——正面偏差可能揭示模型假设过于保守 |

## 示例

### 示例一：性能偏差

**触发**：系统上线后响应时间比设计预期慢 20%。

**偏差量化**：预期 200ms，实际 240ms，偏差率 20%，严重级别，持续恶化。

**五层根因**：慢→某组件处理时间增 50%→输入数据量增 3 倍→算法 O(n²) 但假设了 O(n)→仿真阶段最大数据量仅为实际的 1/10→仿真场景设计遗漏大规模数据场景。

**三级修正**：短期——实现 O(n log n) 算法；中期——仿真场景标准增加"10 倍正常负载"压力测试；长期——简化假设评审增加"假设失效条件"检查。

**认知更新**：原假设"数据量线性增长"修正为"数据量可能非线性增长"，更新至系统思维报告和仿真验证记录。

### 示例二：正面偏差

**触发**：用户留存率比预期高 15%。

**偏差量化**：正向偏差，中等程度，稳定。

**分析结论**：用户画像模型中"核心功能使用频率"的权重设定偏低，实际用户对核心功能的依赖高于预期。

**修正**：中期——调整用户画像模型权重；长期——在模型校准流程中增加"正面偏差假设校验"步骤。
