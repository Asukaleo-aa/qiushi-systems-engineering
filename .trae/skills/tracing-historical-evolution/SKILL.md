# 历史演化分析

## 描述

追溯系统从过去到现在的因果演化链，识别路径依赖性，区分结构必然与偶然遗留，找出历史的未竟之事。核心逻辑：只分析「本质是什么」而忽略「怎么变成这样的」，会错过路径依赖、制度惯性和锁入效应。

## 使用场景

### 何时使用

- 需要理解系统当前状态的形成原因
- 需要判断哪些约束是历史遗留、哪些是结构必然
- 重构老系统前需要理解其演化路径
- 面对路径依赖性约束，需要评估打破的可行性
- 需要区分「一直这样」和「必须这样」

### 何时不使用

- 系统是全新的，没有演化历史
- 历史信息完全不可获取
- 当前问题完全不涉及历史遗留约束
- 只需要知道「它是什么」而不需要理解「为什么长成这样」
- 纯逻辑系统（数学定理、算法），其形态由逻辑必然而非历史偶然决定

## 输入输出

### Input

| 字段 | 类型 | 说明 |
|------|------|------|
| system_description | string | 需要理解演化历史的系统描述 |
| current_problem | string | 当前面临的问题或约束（可选） |
| available_history | list\<string\> | 已知的历史信息来源（可选） |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| evolution_timeline | list\<{stage: string, trigger: string, key_decision: string, legacy: string}\> | 演化时间线，每阶段含出现条件/核心选择/遗留问题 |
| path_dependencies | list\<{lock_point: string, lock_reason: string, alternative_feasibility: string}\> | 路径依赖清单 |
| causation_classification | list\<{item: string, type: "结构必然"\|"情境最优"\|"偶然遗留", basis: string}\> | 因果分类 |
| unfinished_business | {shelved_contradictions: list\<string\>, forgotten_alternatives: list\<string\>, design_debt: list\<string\>} | 历史的未竟之事 |
| evolution_insight | {breakable_constraints: list\<string\>, unbreakable_constraints: list\<string\>, review_points: list\<string\>} | 演化洞察 |

## 指令

1. **建立演化时间线**：按关键转折点将系统划分阶段，每阶段标注出现条件（外部变化+内部未解决问题）、核心选择、遗留问题。不要跳过中间转折点——每个都可能在当前结构上留有印记。
2. **识别路径依赖性**：对每个关键历史决策点，回答三个问题——当时还有其他可行选项吗？后来被质疑过吗？为什么没改？今天重新决策还会选它吗？输出路径依赖清单。
3. **分离偶然与必然**：将历史成因分为三类——结构必然（给定当时条件几乎是唯一选择）、情境最优（当时择优但条件已变）、偶然遗留（纯历史偶然）。每类附判定依据。
4. **找出未竟之事**：识别被搁置的矛盾（早期未解决、可能仍潜伏）、被遗忘的替代路径（历史分叉点被放弃的方向）、积累的设计债（层层打补丁而非重新设计）。
5. **输出演化洞察**：判断哪些历史约束可以打破（锁入原因已消失）、哪些不可打破（结构必然）、哪些值得重新审视。与第一性原理交叉验证——演化路径与本质推导是否一致。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 历史信息不足，无法建立完整时间线 | 先用已有信息建立部分时间线，标注信息缺口；触发 investigating-before-deciding 补充收集 |
| 路径依赖的三问无法回答（缺乏决策记录） | 基于结果反推可能的原因，标注为"推测"而非"确认"；寻找间接证据（代码注释、commit 历史、会议纪要） |
| 偶然与必然难以区分 | 做反事实假设——"如果当时没有这个偶然事件，系统会走向哪里？"通过反事实推理辅助判定 |
| 系统演化阶段过多，分析过于冗长 | 聚焦与当前问题直接相关的最近 2-3 个转折点，更早阶段只记录对当前有持续影响的关键遗留 |
| 演化洞察与第一性原理分析矛盾 | 不急于否定任何一方，标注矛盾点，触发 analyzing-contradictions 分析该元矛盾 |

## 示例

**场景**：一个电商系统的订单模块，当前架构是单体+存储过程，团队想微服务化但发现处处受限。

1. **时间线**：阶段1（创业期）快速上线选单体+存储过程→遗留：业务逻辑锁在存储过程中；阶段2（增长期）加缓存和消息队列补丁→遗留：数据一致性靠存储过程保证，拆不掉；阶段3（当前）想微服务化但存储过程是最大障碍
2. **路径依赖**：锁定点=存储过程承载业务逻辑→锁入原因=创业期追求速度+后来无人重构→替代方案可行性=需先提取存储过程逻辑为应用代码，成本高但可行
3. **因果分类**：单体架构=情境最优（当时速度优先，条件已变）；存储过程承载逻辑=偶然遗留（当时团队DBA主导，非技术必然）；数据一致性约束=结构必然（分布式下一致性是根本挑战）
4. **未竟之事**：被搁置的矛盾=增长期就发现的存储过程可维护性问题；设计债=缓存和消息队列是在单体上打的补丁
5. **洞察**：可打破=单体架构本身；不可打破=数据一致性约束；值得审视=存储过程是否可以渐进式提取而非一次性重写
