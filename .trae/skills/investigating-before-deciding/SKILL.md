# 调查研究

## 描述

一切正确的决策都来源于对实际情况的调查研究。调查不是为既定结论搜集材料，而是让事实来规定判断，让现实来修正理论。本技能提供两种互补模式：哲学模式确保方向正确（不脱离实际、不预设结论），工程模式确保覆盖完整（四层知识地图、多源交叉验证）。

## 使用场景

### 何时使用

- 要做决策但对实际情况了解不够
- 进入陌生领域或面对新问题
- 发现自己在凭"直觉"或"经验"做判断，缺乏事实依据
- 别人的方案听起来合理但不了解其适用条件
- 需要全面了解系统/项目/问题的现状
- 感觉自己在套用理论而不了解实际

### 何时不使用

- 当前任务是已充分调查后的执行阶段——不要重复调查
- 用户已提供完整的上下文和决策依据
- 问题是技术实现细节，答案可直接验证（运行代码看结果）
- 紧急修复任务且问题现象已明确——直接定位修复

## 输入输出

### Input

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| question | string | 是 | 需要弄清楚的核心问题 |
| scope | string | 否 | 调查范围（领域/模块/系统） |
| depth | enum | 否 | 调查深度：philosophy（哲学模式，默认）/ engineering（工程模式） |
| time_constraint | string | 否 | 时间约束 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| current_state | string | 现状描述 |
| key_constraints | string[] | 关键约束条件 |
| new_insights | string[] | 调查前不知道但现在知道的事实 |
| judgment | string | 基于调查的判断（必须先于行动方案出现） |
| knowledge_gaps | string[] | 仍存在的知识缺口（工程模式） |

## 指令

1. **声明调查目的**：一句话写出"我要弄清楚的是：[具体问题]"
2. **制定调查提纲**：列出需要调查的具体问题清单，覆盖以下维度：
   - 当前状况是什么？（现状调查）
   - 历史上是怎样的？（历史调查）
   - 有哪些相关因素？（关联调查）
   - 有哪些不同观点？（多方调查）
3. **深入一线获取第一手资料**：
   - 直接阅读源代码，而非只看文档
   - 直接运行和测试，而非只看报告
   - 直接与相关人员交流，而非只看转述
   - 深入了解细节，不满足于"大概了解"
   - 不得浮光掠影地"走马看花"
4. **详细记录，系统整理**：
   - 记录发现的事实
   - 区分事实与观点
   - 整理归类，找出规律
   - 标注存疑之处
5. **实事求是，得出结论**：
   - 事实先于判断，材料先于观点
   - 结论必须有事实支撑
   - 不得先有结论再找证据
   - 如果事实与预期不符，修改结论而非事实
   - 承认"不知道"也是诚实的调查结论
6. **调查完成后再做决策**：基于调查结果做出决策或提出方案。理论必须服从事实，不是事实服从理论。调查结论必须先于任何行动方案出现，不得边调查边给方案。
7. **输出调查结论**，使用固定格式：
   ```
   调查结论：
   - 现状是：……
   - 关键约束是：……
   - 我之前不知道但现在知道的是：……
   - 基于以上，我的判断是：……
   ```
8. **（工程模式）启用结构化知识地图**：按四层知识地图（哲学层/理论层/方法层/执行层）组织调研，对关键信息标注可靠性标记，执行多源交叉验证。详见 `references/engineering-extension.md`。

## 失败策略

| 失败场景 | 处理方式 |
|----------|----------|
| 不调查就给出答案 | 停止输出方案，先声明调查目的并执行调查提纲 |
| 只看文档不看实际 | 强制要求至少一项第一手验证（读源码/运行测试/直接观测） |
| 调查走马观花、浮于表面 | 对每个调查项追问"具体数据/证据是什么"，拒绝"大概了解" |
| 先有结论再找证据 | 将结论标记为"待验证假设"，重新从事实出发推导 |
| 只调查现状不看历史 | 补充历史维度调查，理解现状的演变路径 |
| 只看一方面 | 补充至少一个不同角度的信息源，兼听则明 |
| 调查后仍无法得出确定结论 | 明确列出知识缺口和待验证假设，承认"不知道"，建议后续行动 |

## 示例

**场景**：用户说「帮我优化这个 API 的性能」，但对 API 内部实现、调用模式、瓶颈一无所知。

1. **声明调查目的**：「我要弄清楚的是：这个 API 的当前性能瓶颈在哪，为什么慢」
2. **调查清单**：
   - [x] 读取 API 源码，了解实现路径（现状）
   - [x] 检查最近的性能监控数据/日志（历史）
   - [x] 运行一次典型请求，观测耗时分布（一手验证）
   - [x] 确认调用量和并发要求（约束条件）
3. **调查结论**：
   - 现状：API 耗时 2.3s，其中数据库查询占 1.8s（N+1 问题）
   - 关键约束：日调用量 50万次，不能增加数据库连接数
   - 新认知：瓶颈不在代码逻辑，在 ORM 的懒加载模式
   - 判断：先优化 N+1 查询（批量预加载），预期降到 300ms
4. **调查结论先于方案**：在给出具体优化代码之前，已明确根因和取舍
