# 第一性原理分析

## 描述

不类比、不套框架、不从"别人怎么做"出发。把问题拆解到不可再分的基本元素，然后从这些元素重新向上推理，构建理解或方案。核心纪律：重建过程中只使用物理约束和逻辑必然，不得引用被搁置的工程惯例、历史遗留和类比借用。

## 使用场景

### 何时使用

- 面对真正新的问题，不知道该用什么方法
- 当前方案都是"参照XX做的"，想找有没有更根本的解法
- 假设已堆积多层，不清楚底层是否还成立
- 需要区分"这个约束是物理上必须的"还是"行业惯例"
- 多个专家给了互相矛盾的建议，需要回到原点重新判断
- 想从源头推导出方案，而非在别人的方案上修补

### 何时不使用

- 问题已有成熟框架且验证有效——直接用，不要为了"本质"去推翻
- 时间紧迫且类比法可以快速给出足够好的解
- 当前处于执行层，不需要重新理解问题本质
- 问题的基本信息尚未收集——先用 `investigating-before-deciding` 摸清现状

## 输入输出

### Input

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| problem | string | 是 | 需要从第一性原理分析的问题 |
| existing_assumptions | string[] | 否 | 当前已知的假设和陈述列表 |
| domain_type | enum | 否 | physical（物理系统）/ product（产品系统），影响约束保留范围 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| stripped_constraints | object[] | 剥离后的约束列表，每项含 {label, content, tag} |
| fundamental_elements | string[] | 不可再分的基本元素 |
| rebuilt_structure | string | 从基本元素重建的结构/方案 |
| cross_check_results | object[] | 与搁置惯例的对碰结果，每项含 {convention, result, insight} |

## 指令

1. **剥离（Strip）**：列出当前关于该问题的所有已知陈述，逐条打标签：

   | 标签 | 含义 |
   |------|------|
   | 物理约束 | 自然规律决定的，不可违反 |
   | 逻辑必然 | 由物理约束+定义推导出的必然结论 |
   | 场景约束 | 由系统使用场景施加的硬边界（谁用、在哪用、怎么用） |
   | 经济约束 | 由系统存续条件（市场、预算、成本结构）施加的边界 |
   | 工程惯例 | 行业内约定俗成但非必然 |
   | 历史遗留 | 过去某个条件下的选择，条件已变 |
   | 类比借用 | "因为XX领域这么做，所以我们也这么做" |
   | 未验证假设 | 可能成立但未经严格检验的前提 |

   物理系统只保留物理约束+逻辑必然；产品系统还需保留场景约束和经济约束。其余标签全部暂时搁置为"可质疑项"。

2. **分解（Decompose）**：将问题分解到不可再分的基本元素：
   - 识别构成问题的最基本的实体
   - 识别实体之间最基本的相互作用
   - 确定描述这些实体和相互作用所需的最基础数学/逻辑形式
   - 判断"到底了"的标准：继续分解不再增加对问题的理解，或分解后的元素已可用一句不含"因为"的话描述

3. **重建（Rebuild）**：从基本元素出发，只使用物理约束和逻辑必然，向上推导：
   - 确定基本元素能组合出什么结构
   - 在给定物理约束下，判断哪些结构可能、哪些不可能
   - 从可能的解中选出对当前目标最优的
   - 重建过程中不得引用第1步被搁置的工程惯例、历史遗留和类比借用——除非重建结果和它们恰好一致

4. **回验（Cross-check）**：将重建结果与搁置的惯例/类比对碰：
   - 一致：惯例有第一性依据，增强信心
   - 不一致：惯例的隐藏前提可能已失效，有创新机会
   - 证明惯例不可能：第一性推导给出了上限，惯例可能错误或条件不同

## 失败策略

| 失败场景 | 处理方式 |
|----------|----------|
| 把"行业最佳实践"当做物理约束 | 重新审视每条约束的标签，物理约束必须是自然规律，行业实践是社会建构，二者严格区分 |
| 分解不够深就停止 | 对每个基本元素追问"能否用不含'因为'的句子描述"，不能则继续分解 |
| 重建时偷渡了之前搁置的假设 | 逐一检查重建推理链中的每个前提，确认其仅来自第一步保留的约束 |
| 对已有成熟理论的问题从头推导 | 成熟理论是前人第一性分析的结果，直接引用，不必重复推导 |
| 混淆产品系统与物理系统的约束保留范围 | 明确 domain_type：物理系统只保留物理约束+逻辑必然，产品系统额外保留场景约束和经济约束 |

## 示例

**场景**：团队讨论「数据库选型应该用 SQL 还是 NoSQL」，每个人都在引用「XX 大厂用的就是这个」。

1. **剥离**：列出所有支持 SQL 和 NoSQL 的理由，逐条打标签——「SQL 是行业标准」→ 行业惯例，「我们需要 JOIN 操作」→ 经验假设，「数据之间有严格关系」→ 物理约束（如果是真的）
2. **分解到基本元素**：数据形状（结构化/半结构化）、查询模式（点查/范围/聚合）、一致性要求（强一致/最终一致）、扩展方向（读/写/存储）
3. **重建**：从四个基本元素出发——结构化数据+强一致+复杂查询→需要关系型；但如果写入量极大且数据结构多变→需要文档型
4. **结论**：不是"应该用哪个"，而是"当前 80% 查询是结构化+需要 JOIN+一致性要求高→关系型数据库。预留 NoSQL 给未来可能的日志和文档存储"
5. **回验**：去掉了"XX 大厂用所以我们也用"的类比推理，保留了基于数据形状和查询模式的结构推理
