# 集中兵力

## 描述

资源有限时，不要分散力量同时应对多个问题，而要集中优势兵力于主要矛盾，彻底解决一个再转向下一个。不打无准备之仗，不打无把握之仗。

## 使用场景

### 何时使用

- 面前有多个任务/问题同时需要处理，资源不足以并行
- 正在同时推进多件事但哪件都做不好
- 需要决定先做什么、后做什么
- 面对一个大问题，需要决定从哪个部分突破

### 何时不使用

- 当前只有一个任务，不存在多任务竞争资源
- 用户明确要求同时推进多个任务（用户指令优先）
- 任务之间完全独立且无资源竞争（如不同文件的独立修改）
- 已在执行选定的主攻任务，不需要重新决策

## 输入输出

### Input

| 字段 | 类型 | 说明 |
|------|------|------|
| tasks | list[Task] | 当前所有待处理任务列表 |
| Task.name | string | 任务名称 |
| Task.impact | enum | 影响范围：high / medium / low |
| Task.difficulty | enum | 解决难度：high / medium / low |
| Task.depended_by | list[string] | 被哪些任务所依赖 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| primary_target | string | 选定的主攻目标名称 |
| rationale | string | 优先此目标的一句话理由 |
| completion_signal | string | 判定该任务完成的具体条件 |
| deferred_tasks | list[{name, reason}] | 暂缓任务及暂缓原因 |

## 指令

1. **盘点所有待处理任务**：为每个任务标注影响范围、解决难度、依赖关系。
2. **确定主要矛盾**：调用 contradiction-analysis，在所有问题中找出那个解决了就能带动其他问题的关键问题。
3. **选择突破口**：按以下优先级选择——
   - 首选：影响范围高 + 被其他任务依赖（解锁阻塞，最高价值）
   - 次选：影响范围高 + 解决难度中（最高性价比）
   - 末选：解决难度低（积累势能，但影响范围不高）
   - 选择原则：先易后难、先孤后强、先关键后外围、有准备才开打。
4. **宣告并锁定主攻目标**：输出主攻目标名称、优先理由、完成信号，以及暂缓任务列表。
5. **集中力量彻底解决**：在解决当前问题期间不分心、不半途而废，追求彻底解决而非临时绕过，在局部问题上追求速决。
6. **验证完成**：确认问题确实被彻底解决，不留后患。
7. **转向下一个目标**：短暂总结经验（调用 criticism-self-criticism），评估剩余问题是否因前序解决而变化，重新确定下一个目标。
8. **不打无准备之仗**：开始每个问题前检查是否充分了解问题、是否有足够资源和可行方案；准备不充分时先调查研究（调用 investigation-first），而非冒然行动。
9. **处理中途新需求**：判断新需求是否属于当前主攻目标范围内——是则纳入，否则记录到待办并说明"当前集中处理 X，完成后立即处理"。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 无法判断哪个任务是主要矛盾 | 调用 investigation-first 补充信息，或选择依赖关系最多的任务作为默认主攻目标 |
| 主攻目标执行中途发现无法完成 | 不立即切换目标，先评估是否需要补充资源或调整方案；确认无法完成后再重新选择突破口 |
| 用户中途提出高优先级新需求打断当前主攻 | 评估新需求紧急程度——紧急则暂停当前任务并记录进度，不紧急则排入暂缓列表 |
| 主攻目标完成后发现解决不彻底 | 回到该目标继续解决，不急于转向下一个，直到完成信号触发 |

## 示例

### 示例一：多 Bug 修复排序

面对 5 个 Bug：B1 影响核心功能且被 B2 依赖、B2 影响边缘功能、B3 影响核心但难度高、B4 影响边缘且难度低、B5 是文档错误。

- 主攻目标：B1（解锁阻塞 + 影响高）
- 完成信号：B1 的测试用例全部通过且 B2 可正常工作
- 暂缓：B2-B5

### 示例二：新项目功能优先级

新项目需要实现：用户认证、数据展示、导出功能、主题切换。资源只够先做一个。

- 主攻目标：用户认证（其他功能均依赖登录状态）
- 完成信号：登录/注册/鉴权流程端到端可用
- 暂缓：其余三项
