# 构建系统边界与结构

## 描述

结构决定功能。子系统划分不合理、接口定义不明确是复杂工程失败的首要原因。本技能对系统进行层次分解、组分规格化、接口契约化和边界六维检查，产出可执行的结构分析报告。

## 使用场景

### 何时使用

- 系统组分超过5个，需要系统化拆解
- 存在多层级嵌套，子系统划分不清晰
- 接口关系复杂，需要明确定义跨组分契约
- 已完成系统思维报告，需要进入精细结构分析

### 何时不使用

- 单体系统，组分数量 ≤ 3
- 结构已经明确且无争议
- 当前任务不需要深入到结构层面
- 用户已提供详细的结构文档

## 输入输出

### Input

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| system_thinking_report | object | 是 | applying-systems-thinking 产出的系统思维报告 |
| focus_areas | string[] | 否 | 需要重点分析的结构维度 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| hierarchy_tree | object | 层次分解树 |
| component_specs | object[] | 每个组分的详细规格 |
| interface_contracts | object[] | 跨组分接口契约 |
| boundary_check | object | 六维边界检查结果 |
| structure_assessment | object | 结构合理性评估 |
| next_actions | string[] | 下一步行动建议 |

## 指令

1. **执行层次分解**——按内聚原则将系统拆分为树状结构：子系统→组件→单元。确保同一子系统内组件功能紧密相关，子系统间接口尽量少且简单，所有组分无遗漏。

2. **定义每个组分的规格**——为每个组分填写统一规格表：功能（一句话）、输入（类型/来源/频率）、输出（类型/去向/频率）、内部状态、配置参数、性能指标、约束条件。

3. **定义接口契约**——为每个跨组分接口定义完整契约：提供方、消费方、数据格式、传输方式、频率、SLA（延迟/吞吐/可用性）、错误处理策略、接口版本号。

4. **执行六维边界检查**——逐一检查物理边界、逻辑边界、时间边界、组织边界、知识边界、场景边界，标记每维度的通过/未通过状态。

5. **评估结构合理性**——从内聚度、耦合度、可扩展性三个维度评估，列出潜在结构问题。

6. **汇总结构分析报告**——整合以上产出，给出下一步行动建议。

详细规格表与接口契约模板见 `references/templates.md`。

## 失败策略

| 失败场景 | 处理方式 |
|----------|---------|
| 层次分解无法确定归属 | 将争议组分暂列为"待定归属"，标注候选归属方，调用 contradiction-analysis 辅助决策 |
| 接口契约无法达成一致 | 先定义最小可行接口，将争议项列为"待协商接口"，标注风险 |
| 六维边界检查多处未通过 | 按严重程度排序，优先修复影响核心功能的维度，其余列入改进计划 |
| 组分规格信息不足 | 标注为"待补充"，调用 investigation-first 收集缺失信息 |
| 结构评估发现严重耦合 | 识别耦合根因，提出解耦方案，必要时回退到步骤1重新分解 |

## 示例

### 示例一：微服务架构拆分

**输入**：一个单体电商应用的系统思维报告，包含订单、商品、用户、支付四个核心模块。

**执行要点**：
1. 层次分解：电商系统→订单服务/商品服务/用户服务/支付服务→各服务内部组件
2. 组分规格：订单服务-功能="管理订单全生命周期"，输入="用户下单事件(异步,用户服务,高频)"，输出="订单状态变更(异步,支付服务+用户服务,中频)"
3. 接口契约：订单→支付接口，同步调用，SLA=延迟≤100ms/可用性99.9%，降级方案=本地记录后异步重试
4. 六维检查：物理边界✓、逻辑边界✓、时间边界✓（未覆盖退役阶段）、组织边界✗（支付团队权责未明确）、知识边界✓、场景边界✓
5. 评估：内聚度高、耦合度中等（订单与支付强耦合）、可扩展性良好

### 示例二：智能制造产线结构化

**输入**：电池包产线的系统思维报告，7个工艺段+质检环节。

**执行要点**：
1. 层次分解：产线→前段(涂布/辊压/分切)/中段(卷绕/注液)/后段(化成/质检)→各工序参数控制单元
2. 接口契约：涂布→辊压，数据格式="膜卷规格+厚度偏差"，传输=同步，SLA="偏差≤2μm时自动放行"
3. 六维检查：知识边界✗（化成工序的参数-性能映射模型尚未建立），场景边界需补充"长时间滑行"工况
4. 评估：内聚度高、耦合度低（工序间通过缓冲区解耦）、潜在问题=化成反馈延迟导致前序参数调整滞后
