# 应用系统思维

## 描述

系统不是部分的简单加和。本技能在复杂任务启动前强制建立系统世界观——定义目的、边界、组分、关联、涌现与反馈，确保后续所有工作建立在系统认知之上，而非盲目执行。

## 使用场景

### 何时使用

- 任务涉及多个子系统或组件，需要整体把握再分解
- 用户目标模糊，需要澄清系统使命与成功标准
- 涉及多个利益相关方或相互制约的约束条件
- 需要长期规划或多组件协调的工程或研究问题
- 进入陌生领域，需要先建立全局认知框架

### 何时不使用

- 任务范围极小（修改一行代码、回答简单问题）
- 任务已在已有系统框架内，当前步骤是纯执行
- 用户明确要求"直接做，不要想太多"
- 紧急修复任务且问题边界完全明确

## 输入输出

### Input

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| task_description | string | 是 | 任务的原始描述 |
| known_constraints | string[] | 否 | 已知的硬约束列表 |
| stakeholders | string[] | 否 | 已识别的利益相关方 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| system_mission | string | 一句话系统使命 |
| success_metrics | string[] | 可量化的成功标准 |
| boundary | object | 系统内/外/交换接口定义 |
| component_relations | object | 组分关联矩阵 |
| hierarchy | object | 四层分解结构 |
| emergent_properties | object | 预期与潜在意外涌现 |
| feedback_loops | object | 正/负反馈回路与延迟 |
| unknowns | string[] | 关键未知项 |
| next_actions | string[] | 后续行动建议 |

## 指令

1. **判定系统层次**——识别当前系统属于物理系统、工程系统、产品系统还是社会-技术系统。若有终端用户则为产品系统；若为更大系统的子组件则目标由母系统分解得出；仅自封闭系统可用内部指标作为目标。

2. **用一句话定义系统使命**——使命必须与系统层次一致。列出可量化的成功指标和不可违反的硬约束。

3. **划定系统边界**——区分系统内（可控组分）与系统外（环境、不可控条件），明确系统与环境之间的输入、输出、约束交换。

4. **识别组分与关联**——列出所有子系统/组分，标注组分间的作用类型（依赖、制约、增强、反馈），构建关联矩阵区分关键路径与次要关联。

5. **分解层次结构**——按哲学层（根本假设）、理论层（核心原理）、方法层（技术工具）、执行层（操作流程）四层分解系统。

6. **发现涌现特性**——识别单独组分不具备但组合后出现的特性，区分预期涌现与潜在意外涌现，标注触发条件。

7. **建立反馈结构**——识别正反馈（放大）与负反馈（稳定）回路，标注关键延迟时间。

8. **汇总系统思维报告**——将以上七步产出整合为完整报告，包含关键未知项与后续行动建议。

详细输出格式模板见 `references/output-formats.md`。

## 失败策略

| 失败场景 | 处理方式 |
|----------|---------|
| 无法确定系统层次 | 默认按产品系统处理，标注为假设，后续验证 |
| 系统边界模糊、内外难以区分 | 先按最窄边界定义，将不确定项列入"待确认边界项"，调用 investigation-first 调研 |
| 组分关联过于复杂、矩阵无法收敛 | 聚焦关键路径，将次要关联降级为备注，调用 structuring-system-boundaries 做精细分析 |
| 涌现特性无法预判 | 仅列出高置信度的预期涌现，其余归入"关键未知项" |
| 用户拒绝系统思维、要求直接执行 | 输出最简版系统使命+边界（不超过5行），标注风险后遵从用户意图 |

## 示例

### 示例一：电商平台重构

**输入**：用户要求重构一个电商平台，当前系统响应慢、模块耦合严重。

**执行要点**：
1. 判定层次：产品系统（有终端用户），母系统为公司业务体系
2. 定义使命："在保持业务连续性的前提下，将系统响应时间降至 200ms 以内，并实现核心模块解耦"
3. 划定边界：系统内=订单/商品/用户/支付四个模块；系统外=第三方支付/物流/监管合规
4. 识别关联：订单→依赖→商品+用户+支付；商品→制约→库存
5. 层次分解：哲学层=用户体验优先；理论层=微服务架构；方法层=DDD 领域建模；执行层=逐模块拆分迁移
6. 涌现：预期=独立部署能力；意外=分布式事务一致性风险
7. 反馈：正反馈=用户增长→数据增长→推荐精准→用户增长；负反馈=负载升高→限流→负载降低

### 示例二：智能制造产线设计

**输入**：设计一条新能源汽车电池包的智能制造产线，涉及多个工艺段和质检环节。

**执行要点**：
1. 判定层次：工程系统（物理系统+制造可行性），母系统为产品系统
2. 定义使命："实现电池包产线节拍 ≤ 60s/台，良品率 ≥ 99.5%"
3. 划定边界：系统内=涂布/辊压/分切/卷绕/注液/化成/质检；系统外=原材料供应/环保法规/市场需求
4. 识别关联：涂布→制约→辊压→制约→分切；质检→反馈→各工艺段
5. 涌现：预期=产线整体节拍由瓶颈工序决定；意外=某工序良率波动通过缓冲区放大
6. 反馈：负反馈=质检不合格→前序参数调整→良率回升；延迟=化成工序耗时导致反馈滞后数小时
