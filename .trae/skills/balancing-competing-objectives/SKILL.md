# 统筹兼顾

## 描述

复杂系统中存在多对辩证关系，不能只顾一头而忽略另一头。本技能识别所有重要的关系对，在它们之间找到动态平衡，最大限度地调动一切积极因素——不是各方都少一点的妥协，而是让各方面都能发挥积极作用的最优配置。

## 使用场景

### 何时使用

- 面对多个相互制约的目标，需要权衡取舍
- 发现优化一个维度时损害了另一个维度
- 系统中存在多种利益相关方，需要协调
- 在速度与质量、深度与广度、短期与长期等对立面之间做选择
- 发现自己正在片面地追求某一个目标

### 何时不使用

- 只有一个目标维度，不存在需要平衡的对立面
- 用户已明确取舍方向（用户决策优先）
- 大方向已定，只需集中推进——用 `concentrate-forces`
- 问题的性质是"必须选一个放弃另一个"的硬性约束，不存在动态平衡空间

## 输入输出

### Input

| 字段 | 类型 | 说明 |
|------|------|------|
| objectives | list[str] | 当前涉及的所有目标或利益方 |
| constraints | list[str] | 系统性约束条件 |
| context | str | 当前阶段、资源状况等背景信息 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| dialectical_pairs | list[object] | 识别出的辩证关系对及当前偏向 |
| balance_plan | list[object] | 每对关系的当前阶段平衡点与最低限度 |
| impact_matrix | list[object] | 调整一对关系对其他关系的影响 |
| warning_indicators | list[str] | 失衡预警信号 |

## 指令

1. **识别所有辩证关系对**：列出当前系统/任务中所有重要的对立面（如速度↔质量、短期↔长期、深度↔广度），标注当前偏向与偏向程度。不同任务的辩证关系对不同，根据具体情况识别，不照搬模板。

2. **检查片面性**：对每对关系，判断当前方案是否完全忽略了其中一方。区分刻意的取舍（可接受）与无意的忽略（需纠正）。

3. **确定当前阶段的平衡点**：为每对关系确定优先侧与另一方的最低限度，说明原因。平衡不是固定折中，根据具体条件动态调整。

4. **检查系统影响**：调整一对关系后，检查对其他关系的影响（加剧/缓解/无影响），必要时联动调整。

5. **参考他人教训**：查找类似场景中的失败案例（过度优化某一维度的陷阱），借鉴但不照搬，结合自身实际调整。

6. **寻找积极因素配置**：探索让两个看似矛盾的目标同时推进的可能性，挖掘被忽视的资源或方法，将负面因素转化为正面因素。

7. **设置失衡预警指标**：为每对关系定义具体信号，当信号出现时重新调用本技能调整平衡。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 关系对识别不全，遗漏重要对立面 | 回到目标清单，逐项检查"优化它是否可能损害其他目标"，补充遗漏的关系对 |
| 平衡点固定不变，未随条件变化调整 | 引入阶段性复盘机制，每完成一个里程碑重新评估平衡点 |
| 各方妥协式"和稀泥"，没有调动积极因素 | 检查是否只做了减法分配，尝试寻找创造性方案让多方同时受益 |
| 调整一处引发连锁失衡 | 回退调整，改用增量式微调并逐项检查全局影响 |
| 照搬他人方案导致水土不服 | 标注"借鉴来源"与"本地差异"，逐项验证适用性 |

## 示例

### 示例一：项目交付的速度与质量平衡

**情境**：团队需要在2周内交付一个功能模块，但代码质量要求高。

1. 识别关系对：速度 ↔ 质量，当前偏向速度（严重）
2. 片面性检查：当前计划完全跳过了代码审查环节，属于无意忽略
3. 平衡点：当前阶段优先速度，但确保核心模块通过代码审查（最低限度）
4. 系统影响：加强质量检查会延长1天工期，但减少返工风险，对总工期影响可控
5. 预警指标：如果返工率超过20%，说明质量投入不足，需重新调整

### 示例二：产品功能的深度与广度取舍

**情境**：产品需要同时覆盖多个用户场景，但每个场景的深度不够。

1. 识别关系对：深度 ↔ 广度，当前偏向广度（明显）
2. 片面性检查：所有场景都只做了表面功能，核心场景无差异化竞争力
3. 平衡点：先在2个核心场景深入，其余场景保持最小可用
4. 积极因素：深入场景的经验可复用到广度扩展，形成"深度驱动广度"的路径
5. 预警指标：如果核心场景用户留存率低于预期，说明深度不足需继续投入
