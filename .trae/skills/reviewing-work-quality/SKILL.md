# 审视工作质量

## 描述

定期的、诚实的自我审视和相互批评是防止错误积累的唯一机制。批评的目的不是惩罚，而是"治病救人"——每指出一个问题就提出一个改进建议，关注"下次怎么做更好"而非"这次谁的错"。

## 使用场景

### 何时使用

- 完成一项工作后，需要审视其质量
- 接收到他人的批评或反馈
- 发现自己的工作可能有问题但不确定
- 项目进行到阶段性节点，需要复盘
- 反复出现同类错误，需要找到根本原因
- 需要为团队或项目建立定期审查机制

### 何时不使用

- 任务刚开始，还没有可以审视的工作成果
- 用户在实时指导中，反馈已经即时纳入
- 简单的单步任务（如查一个函数签名）
- 已处于紧急修复的执行阶段（先完成修复，结束后再审视）

## 输入输出

### Input

| 字段 | 类型 | 说明 |
|------|------|------|
| work_description | string | 待审视的工作内容和原定目标 |
| acceptance_criteria | list | 验收标准 |
| external_feedback | list | 收到的外部批评或反馈（可为空） |
| recurring_issues | list | 反复出现的同类问题（可为空） |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| goal_status | object | 原定目标完成情况（已完成/未完成及原因） |
| issues | list | 发现的问题（含严重程度、具体描述、根本原因、改进建议） |
| strengths | list | 做得好的地方 |
| next_focus | list | 下次重点关注事项 |

## 指令

1. **先自我批评**：在请别人审查之前，先诚实审视自己的工作——目标完成了吗？哪些地方做得不够好？有没有偷懒、敷衍、走捷径？有没有知道有问题但选择忽略的？如果重新来过会在哪里做不同选择？不讳疾忌医，自己已知的问题必须主动暴露。

2. **具体而非笼统**：指出具体的行为或决策而非笼统评价，分析错误产生的原因而非只描述现象，说明错误造成的影响，提出改进的具体方法。禁止写"做得不够好"这类无信息量描述，必须包含具体步骤编号或文件名。

3. **基于事实而非猜测**：引用具体的事实和证据，不依据传言或猜测，对不确定的部分标注"存疑"，允许被批评方提供补充信息。

4. **治病救人而非整人**：每指出一个问题就提出一个改进建议，承认问题的同时肯定做得好的部分，关注"下次怎么做更好"而非"这次谁的错"，把错误视为学习机会。

5. **欢迎他人批评**：先完整听取不急于辩解，判断批评是否基于事实——正确则立即接受改正，不准确则用事实说明而非情绪化回应，感谢提出批评的人。

6. **形成常态机制**：每完成阶段性工作就进行一次自我审视，定期复盘而非出了问题才反思，把上次发现的问题作为下次检查重点，积累错误记录找出模式和根本原因。

7. **输出结构化审视报告**：包含原定目标、完成情况、发现的问题（表格：严重程度/问题描述/根本原因/改进建议）、做得好的地方、下次重点关注。若问题表格为空，必须写出"本次工作未发现需要改正的问题，原因是……"。将"下次重点关注"输出给用户，并告知可将其作为下次任务的上下文。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 自认为完美，自查无问题 | 强制过检查清单：是否跳过步骤？是否充分调查？是否实践验证？是否抓住主要矛盾？是否有回避的问题？ |
| 批评笼统不具体 | 要求重新审视，每个问题必须包含：具体行为/决策 + 产生原因 + 造成影响 + 改进方法 |
| 表面接受批评实际不改 | 将改进建议转化为可验证的具体行动项，下次审视时检查上次的行动项是否落实 |
| 批评变成追责或人身攻击 | 立即纠正方向：批评针对具体行为而非个人，每个问题必须附带建设性改进建议 |
| 只在出问题时才反思 | 建立定期审视节奏（每完成一个阶段性工作即审视），将审视从应急措施变为日常机制 |

## 示例

### 示例一：阶段工作复盘

完成一个功能模块开发后的审视。原定目标：实现用户认证模块，通过安全审计。完成情况：功能已实现，安全审计未通过（原因：跳过了输入校验环节）。发现问题：严重程度"必须改正"——第三步跳过了输入校验，根因是时间压力下选择了走捷径，改进建议是在开发流程中加入校验清单强制检查。做得好的地方：接口设计清晰。下次重点关注：不得在时间压力下跳过安全检查步骤。

### 示例二：收到外部批评

用户反馈"生成的代码缺少错误处理"。审视：检查最近交付的三个模块，发现确实有两个模块的错误处理不完整。根因：开发时只关注了正常路径，未系统考虑异常场景。改进建议：每个模块交付前用异常场景清单逐项检查。下次重点关注：交付前必须过异常场景清单。
