# 矛盾分析法

## 描述

识别问题中的所有对立面，隔离出规定其他矛盾的主要矛盾，判定矛盾是对抗性还是非对抗性，并据此选择解决路径。核心逻辑：捉住主要矛盾，一切问题迎刃而解。

## 使用场景

### 何时使用

- 面对复杂问题，不知道从哪里下手
- 问题中存在多个对立面或冲突因素
- 需要判断多个问题中哪个最重要、最紧迫
- 试图解决一个问题但总是被其他问题干扰
- 处理团队/系统中的内部冲突或外部冲突

### 何时不使用

- 任务是直接执行性的（如格式化代码），没有矛盾需要分析
- 用户已明确指定解决方案，只需实现
- 问题只有一个维度，不存在对立面
- 时间紧急且问题属于已知模式，直接套用已验证方案

## 输入输出

### Input

| 字段 | 类型 | 说明 |
|------|------|------|
| problem_context | string | 当前面临的问题情境描述 |
| known_forces | list\<string\> | 已知的冲突因素或对立面（可选） |
| constraints | list\<string\> | 时间、资源等约束条件（可选） |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| contradiction_list | list\<{opposing: [string, string], note: string}\> | 所有识别到的矛盾，格式为 [A] vs [B] |
| principal_contradiction | {opposing: [string, string], reason: string} | 主要矛盾及判定理由 |
| nature | "对抗性" \| "非对抗性" | 主要矛盾的性质 |
| resolution_method | string | 对应的解决方法及下一步行动 |
| monitor_hint | string | 需要监控的矛盾转化风险 |

## 指令

1. **列出矛盾清单**：识别当前情境中所有对立的、互相制约的因素，每对矛盾明确其两个对立面，格式为 `[A] vs [B]`。不要只看表面现象，深入分析内在张力。
2. **判定主要矛盾**：在所有已识别的矛盾中，找出那个规定或影响其他矛盾的矛盾。判断标准：解决了它，其他矛盾是否随之缓解？它是否是其他矛盾产生的根源？它是否处于当前阶段的中心位置？标记为 `⭐ 主要矛盾`，用一句话说明理由。
3. **分析矛盾的主要方面**：在主要矛盾内部，判断哪一方面占支配地位，这决定事物的当前性质和发展方向。
4. **区分矛盾性质**：判定主要矛盾是对抗性还是非对抗性。对抗性矛盾（根本利益冲突）需明确立场、果断取舍；非对抗性矛盾（共同利益下的分歧）可协商、求同存异。
5. **选择解决方法**：根据矛盾性质选择对应路径——对抗性矛盾果断决策；非对抗性矛盾民主讨论；技术矛盾调查研究；资源矛盾集中兵力；多维平衡统筹兼顾；多源信息矛盾综合集成。明确写出"接下来我将……"。
6. **设置矛盾转化监控**：在分析末尾标注需监控的风险——主要矛盾是否可能转化？非对抗性矛盾是否可能因处理不当激化为对抗性？

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 无法识别出任何矛盾 | 退回 investigating-before-deciding 补充事实；若确实无矛盾，说明本技能不适用并退出 |
| 多个矛盾看似同等重要，无法判定主次 | 先按影响范围排序，选择影响面最广的作为暂定主要矛盾，在后续实践中验证和调整 |
| 矛盾性质判定模糊（对抗/非对抗边界不清） | 先按非对抗性处理，避免激化；设置更密集的监控点，一旦发现激化迹象立即切换策略 |
| 解决主要矛盾后次要矛盾上升为新主要矛盾 | 触发 feedback-and-revision-loop，重新执行矛盾分析流程 |
| 非对抗性矛盾被错误地按对抗性处理导致激化 | 立即暂停，重新评估矛盾性质，启动批评与自我批评流程修复关系 |

## 示例

**场景**：接手遗留项目，用户抱怨「功能开发太慢，bug 太多，架构需要重构」，三个问题互相牵制。

1. **矛盾清单**：开发速度 vs 代码质量；新功能 vs 重构；用户期望 vs 技术债务
2. **主要矛盾**：⭐ 代码质量 vs 开发速度——质量差导致每次加功能都引入新 bug，修复又拖慢开发，质量差是根源
3. **性质**：非对抗性——质量提升最终会让开发加速
4. **解决方法**：每次开发新功能时强制提升相关模块质量，渐进式改善，而非「先重构再开发」或「不管质量赶功能」
5. **监控**：⚠️ 需监控：用户期望 vs 技术债务 是否因渐进式改善见效慢而上升为主要矛盾
