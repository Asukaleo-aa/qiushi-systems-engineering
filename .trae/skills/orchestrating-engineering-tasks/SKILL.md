# 工程编排与协调

## 描述

再好的设计，如果执行不力也是空谈。本技能将系统思维转化为可操作的任务序列——通过工作分解、依赖分析、关键路径识别和质量门设置，确保复杂系统的实施有序推进、风险可控。

## 使用场景

### 何时使用

- 系统设计已完成，需要制定执行计划
- 任务数量多且存在依赖关系，需要编排调度
- 涉及多个团队或 agent 协作，需要协调资源
- 需要识别关键路径和瓶颈以控制工期

### 何时不使用

- 任务极其简单（少于3个步骤且无依赖）
- 系统设计尚未完成或未通过验证
- 纯探索/研究性质的工作（无明确交付目标）
- 一次性操作且不需要重复执行

## 输入输出

### Input

| 字段 | 类型 | 说明 |
|------|------|------|
| system_design | object | 已验证的系统设计（含子系统/组件结构） |
| constraints | list[str] | 资源、时间、技术等约束条件 |
| stakeholders | list[str] | 参与方（人/agent/团队）及其能力 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| wbs | object | 工作分解结构（树状任务 + 规格表） |
| dependency_graph | object | 依赖关系图（含依赖类型标注） |
| critical_path | list[str] | 关键路径及总工期估算 |
| phases | list[object] | 阶段划分（含准入/退出条件与质量门） |
| risk_matrix | list[object] | 风险矩阵（概率/影响/缓解/应急） |
| execution_order | list[object] | 按优先级排序的任务列表（含负责方与截止时间） |

## 指令

1. **分解工作结构**：从系统结构映射到工作任务（SBS → WBS），逐层分解至可执行粒度。每个任务必须包含：描述、负责方、输入（前置产出物）、产出物、预计工作量、验收标准。

2. **分析依赖关系**：使用前驱图分析任务依赖，标注依赖类型（FS/SS/FF/SF），区分串行任务、并行任务与双向依赖任务。

3. **识别关键路径**：计算每条路径的总时间，找出决定总工期的关键路径。关键路径上的任何延迟直接拖累整体工期。

4. **规划执行阶段**：按规划→设计→开发→集成→验证→交付划分阶段，每个阶段定义准入条件、退出条件和检查点（Gate Review）。设置质量门：方向正确性→设计充分性→组件通过性→集成通过性→目标满足性。

5. **识别与缓解风险**：建立风险矩阵（概率×影响），重点关注关键路径风险、耦合点风险和知识缺口风险。为每个高风险项制定缓解措施和应急预案。

6. **调度资源**：分析资源负载，识别资源争抢，确保关键资源集中在关键路径上。

7. **输出编排计划**：按固定格式输出 WBS、依赖关系、关键路径、阶段规划、风险矩阵、执行命令和监控指标。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| WBS 分解粒度过粗，任务无法执行 | 回到系统结构，按组件→模块→函数逐层细化，直到每个任务有明确的验收标准 |
| 关键路径识别遗漏，实际工期远超预期 | 补充隐式依赖（如知识传递、环境准备），重新计算路径 |
| 质量门被跳过，缺陷累积到后期 | 强制执行门禁：未通过验收标准的任务不得进入下一阶段，缺陷修复优先级高于新功能开发 |
| 资源争抢导致关键路径阻塞 | 优先保障关键路径资源，非关键路径任务可延迟或重新分配 |
| 风险未提前识别，发生时无预案 | 对已发生风险启动应急响应，补充风险识别流程（每个 Gate 增加风险复查环节） |

## 示例

### 示例一：前后端分离项目的工程编排

**情境**：一个 Web 应用包含前端、后端 API 和数据库三个子系统。

1. WBS：前端（3个模块）、后端（4个模块）、数据库（2个模块）、集成测试
2. 依赖：前端模块依赖后端 API（FS），后端模块依赖数据库 Schema（FS），集成测试依赖全部完成（FS）
3. 关键路径：数据库Schema→后端核心API→集成测试，总工期14天
4. 质量门：Gate1（Schema评审）→ Gate2（API单元测试通过）→ Gate3（集成测试通过）→ Gate4（验收）
5. 风险：API接口变更影响前端（概率中/影响高）→ 缓解：接口契约先定稿，变更走评审流程

### 示例二：多 agent 协作的数据管道开发

**情境**：数据采集、清洗、分析三个 agent 需要协作完成数据管道。

1. WBS：采集（2任务）、清洗（3任务）、分析（2任务）、端到端验证
2. 依赖：清洗依赖采集输出（FS），分析依赖清洗输出（FS），采集与分析可部分并行（SS）
3. 关键路径：采集→清洗核心逻辑→分析→端到端验证
4. 资源调度：清洗 agent 负载最重，安排在关键路径上优先执行
5. 预警指标：如果采集延迟超过1天，触发清洗任务重排
