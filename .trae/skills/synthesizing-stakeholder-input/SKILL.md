# 群众路线

## 描述

正确的认识和决策来自双向的知识流动：从实践者那里收集分散的意见，经过系统化整理后返回给实践者验证和执行，如此循环往复，一次比一次更正确。在 AI 语境中，"群众"指一切持有真实情况的信息源——代码库、日志、文档、用户反馈等。

## 使用场景

### 何时使用

- 需要收集用户/使用者/相关方的意见和需求
- 制定方案后需要征求多方反馈
- 感觉在"闭门造车"，脱离实际使用者需求
- 需要整合多个不同角度的意见形成统一方案
- 方案执行效果不佳，需要回到使用者重新收集意见

### 何时不使用

- 任务信息来源单一且完整（用户提供了所有必要上下文）
- 已完成充分的多源信息收集，处于执行阶段
- 紧急修复且问题来源明确——不需要多源汇聚
- 用户是唯一信息源且其输入已足够精确（如"修改第3行的变量名"）

## 输入输出

### Input

| 字段 | 类型 | 说明 |
|------|------|------|
| information_sources | list[str] | 可用的信息源（用户、代码库、日志、文档等） |
| question | str | 需要收集意见的核心问题 |
| existing_plan | str | 已有方案（如有，用于返回验证） |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| collected_facts | list[object] | 从各信息源提取的关键事实 |
| synthesis | object | 系统化整理结果（一致处/矛盾处/缺口） |
| judgment | str | 综合判断及依据 |
| validation_plan | object | 验证方案（如何返回给实践者检验） |

## 指令

1. **识别信息源**：列出所有可用的信息源——用户的明确需求、代码库现状、运行结果/日志、相关文档/注释/git history。在 AI 语境中，"群众"即一切持有真实情况的信息源。

2. **收集事实**：从各信息源提取关键事实（不是解释，是事实）。收集到的意见通常是分散的、互相矛盾的——先全面收集，不急于评判或筛选。主动去问，不被动等待。

3. **系统化整理**：识别各信息源之间的一致之处（可作为可信事实）、矛盾之处（调用矛盾分析处理）、信息缺口（标注需进一步调查）。系统化不是简单多数投票，要运用分析能力提炼零散智慧。

4. **形成综合判断**：用固定格式输出——综合判断、判断依据（哪些信息源共同指向）、与用户原始描述有出入的地方。

5. **返回验证**：将系统化方案返回给实践者/使用者，用他们能理解的方式解释，说明方案如何从他们的意见中提炼，征求反馈。让他们感到这是"自己的意见被整理了"而非"被强加的方案"。

6. **在实践中检验**：方案执行后观察效果——是否解决原问题、使用者是否觉得合理可行、有无预料之外的问题。

7. **开启新一轮循环**：根据实践检验结果重新收集意见。每轮循环应比上一轮更准确、更有效、更深入。

8. **个别指导先行**：执行方案时，先在一个具体点深入实施取得经验，用此经验验证和丰富一般性方案，再推广到更大范围。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 陷入命令主义——自己制定方案不征求使用者意见 | 强制执行信息源清单检查，确认每个信息源都已采集后再输出方案 |
| 陷入尾巴主义——用户说什么做什么，不加分析 | 对矛盾意见必须调用矛盾分析，输出时标注"专业判断与用户意见的差异" |
| 只收集不验证，方案未在实践中检验 | 在方案输出中强制包含验证计划，设定验证截止时间，到期检查结果 |
| 一次定案不迭代 | 在方案中预设复盘节点，每个节点触发新一轮收集-集中-返回循环 |
| 只听一方意见，信息源单一 | 检查信息源清单是否覆盖至少3个独立来源，不足时补充调查 |

## 示例

### 示例一：功能需求的群众路线验证

**情境**：用户要求添加一个导出功能，但需求描述模糊。

1. 信息源：用户描述、现有代码中的导出相关逻辑、错误日志中的导出失败记录、产品文档中的导出规格
2. 收集事实：用户说"导出Excel"，代码中已有CSV导出，日志显示大文件导出超时，文档要求支持3种格式
3. 系统化：一致处——需要导出功能；矛盾处——用户只要Excel但文档要求3种格式；缺口——大文件性能要求不明
4. 判断：先实现Excel导出（用户核心需求），预留多格式扩展点，大文件采用流式导出解决超时
5. 返回验证：向用户确认Excel优先级、流式导出的交互方式
6. 验证：实现后测试10万行数据导出，确认不超时

### 示例二：技术方案的多源交叉验证

**情境**：需要选择缓存方案，团队意见分歧。

1. 信息源：团队A主张Redis、团队B主张Memcached、现有代码中的缓存使用模式、性能测试数据
2. 收集事实：团队A需要复杂数据结构、团队B只需简单KV、代码中80%是KV操作、性能测试两者差异<5%
3. 系统化：矛盾处——数据结构需求分歧；一致处——性能要求相近
4. 判断：默认用Memcached满足80%场景，复杂场景用Redis，通过抽象层统一接口
5. 返回验证：将方案提交两个团队评审，确认抽象层接口满足各自需求
6. 检验：先在一个模块试点，验证性能和开发体验
