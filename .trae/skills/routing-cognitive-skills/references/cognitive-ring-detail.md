# 七环认知环详解

## 认知环结构

```
         ┌──────────────────────────────────────────┐
         │              实事求是（总原则）             │
         └──────────────────────────────────────────┘
                            │
    ┌───────────────────────┼───────────────────────┐
    ▼                       ▼                       ▼
┌─────────┐           ┌─────────┐           ┌─────────┐
│  感 知  │ ────────→ │  理 解  │ ────────→ │  建 模  │
│ Perceive│           │Understand│           │  Model  │
└─────────┘           └─────────┘           └─────────┘
    ▲                                               │
    │                                               ▼
    │                                          ┌─────────┐
    │                                          │  求 解  │
    │                                          │  Solve  │
    │                                          └─────────┘
    │                                               │
    │                                               ▼
    │   ┌─────────┐    ┌─────────┐    ┌─────────┐
    └───│  修 正  │◄───│  校 验  │◄───│  执 行  │
        │ Correct │    │ Verify  │    │ Execute │
        └─────────┘    └─────────┘    └─────────┘
```

## 调度规则

仅在以下任一条件成立时调用下游 skill：
- 任务明显匹配某个 skill 的触发条件
- 当前结果质量会因为该 skill 明显提升
- 已遇到阻塞，需要一个明确的方法框架来推进

每个认知环节内，多个 skill 之间存在协同关系——不是"选一个"，而是"确认哪几个需要同时上场"。

---

## 第一环：感知（Perceive）

核心命题：我面对的是什么？已知什么，未知什么？现状是什么？

| 认知子任务 | 可调 skill | 协同规则 |
|-----------|-----------|---------|
| 获取一手事实，摸清基本情况 | `investigation-first` | 先调用。定义观测框架：看什么、问谁、怎么验证 |
| 进入陌生领域需要系统性知识梳理 | `investigation-first`（工程扩展模式） | 启用结构化知识地图 + 多源验证 |
| 从噪声、矛盾、残缺数据中推断隐藏状态 | `state-estimation` | 前置：investigation-first 定义观测变量后，用 state-estimation 做递推更新 |
| 收集多方意见，整合利益相关方反馈 | `mass-line` | 用于"人"维度的信息收集，与工程调研互补 |

**出口条件：**
- 已知什么（事实清单）
- 未知什么（信息缺口清单）
- 哪些信息源可靠，哪些不可靠

---

## 第二环：理解（Understand）

核心命题：这个东西的本质是什么？主要矛盾是什么？判断建立在什么前提之上？

| 认知子任务 | 可调 skill | 协同规则 |
|-----------|-----------|---------|
| 剥离假设到不可再分的基本元素 | `first-principles-analysis` | 在感知数据基础上做。先剥离（什么是物理约束、什么是惯例），再分解，再重建 |
| 追溯系统如何演化为当前状态 | `historical-evolution-analysis` | 与第一性原理分析互补：前者问「本质是什么」，后者问「怎么变成这样的」 |
| 找到核心张力，确定主攻方向 | `contradiction-analysis` | 前置：第一性原理分析和历史演化分析完成后调用。主要矛盾 = 约束瓶颈 = 优先攻克的目标 |

**出口条件：**
- 问题的本质用一句话说清
- 系统的演化路径和关键转折点被识别
- 主要矛盾被明确标注，附带判定依据
- 关键假设被列出，每个标注了验证状态

---

## 第三环：建模（Model）

核心命题：怎么描述这个系统？组成部分、边界、关系、可计算形式？

| 认知子任务 | 可调 skill | 前置依赖 |
|-----------|-----------|---------|
| 建立系统世界观：边界、组分、关联、涌现 | `systems-thinking-framework` | 理解环节的产出（本质 + 主要矛盾） |
| 精细划分子系统、定义接口契约 | `system-boundary-structuring` | 系统思维框架的粗粒度分解完成后 |
| 将定性认识转化为可计算模型 | `quantitative-modeling-workflow` | 系统结构分析完成后 |
| 定性判断 + 定量结果 + 经验 → 统一结论 | `meta-synthesis-engine` | 各局部分析产出汇聚时 |

**出口条件：**
- 系统有明确的边界和组分分解
- 关键关系被量化或至少被定性描述
- 模型的适用范围和局限性被声明

---

## 第四环：求解（Solve）

核心命题：在系统模型内，面对约束，最优方案是什么？

| 认知子任务 | 可调 skill | 使用时机 |
|-----------|-----------|---------|
| 在约束下找到最优决策 | `constrained-optimization` | 目标明确、行动空间受限时 |
| 将大系统分解为自治子系统并协调 | `hierarchical-decomposition-coordination` | 多子系统互扰、局部优化恶化全局时 |
| 正向求解受阻，构造对偶问题 | `duality-complementarity-analysis` | 正向路径卡住、问题有对称结构时 |
| 精确求解不可行，逐步逼近 | `perturbation-progressive-method` | 直接求解不现实，需要从简单情形逼近时 |
| 在当前数学工具下卡住，切换表述 | `multi-representation-equivalence-transform` | 同一方向反复尝试无果，换个数学语言看 |

**出口条件：**
- 至少得到一个可行解（不必是最优解）
- 明确标注解的适用范围和敏感假设
- 标出瓶颈约束（哪个约束限制了进一步优化）

**非数学化翻译规则**：当问题结构匹配某个求解 skill 的模式但问题本身不是数学问题时：

1. **定性映射**——把非数学问题映射到该 skill 的数学结构
2. **方法论求解**——用该 skill 的方法论在映射后的结构中求解
3. **翻译回原场景**——将解翻译为原场景中的可执行行动
4. **声明映射**——在结论前声明做了什么映射

---

## 第五环：执行（Execute）

核心命题：怎么把方案变成有序的、可追踪的行动？

| 认知子任务 | 可调 skill | 协同规则 |
|-----------|-----------|---------|
| 将方案编排为有依赖关系的任务序列 | `engineering-orchestration` | 先调用。产出 WBS + 关键路径 + 质量门 |
| 从编排计划中选择当前主攻方向 | `concentrate-forces` | 前置：编排完成后，在关键路径上确定"当前战役目标" |
| 从零起步，建立最小可行根据地 | `spark-prairie-fire` | 当资源极度有限、需要先验证可行性时优先调用 |
| 分阶段布局长期复杂任务 | `protracted-strategy` | 当短期无法速胜、需要划分防御/相持/反攻阶段时调用 |
| 多个目标相互制约时找动态平衡 | `overall-planning` | 当优化一个目标会伤害另一个时，在集中兵力之前先做全局平衡 |

**出口条件：**
- 有明确的任务依赖图
- 当前的"主攻目标"被宣告
- 有可检查的里程碑和质量门

---

## 第六环：校验（Verify）

核心命题：结果符合预期吗？偏差在哪？为什么？

| 认知子任务 | 可调 skill | 协同规则 |
|-----------|-----------|---------|
| 实践→认识→再实践的螺旋验证 | `practice-cognition` | 全环节的方法论约束——每个环节都应受其约束 |
| 多场景仿真，边界条件测试 | `simulation-validation-cycle` | 当模型已建立、需要测试极端情况时调用 |
| 阶段完成后系统检测偏差 | `feedback-and-revision-loop` | 每个执行阶段完成后调用，量化偏差并追溯根因 |

**出口条件：**
- 实际结果与预期的偏差被量化
- 偏差的根因被追溯到所属层次（执行层 / 模型层 / 理解层 / 感知层）
- "这个偏差是噪声还是真问题"的判断被给出

---

## 第七环：修正（Correct）

核心命题：基于偏差，改什么？方案？模型？理解？还是从头再感知？

| 认知子任务 | 可调 skill | 协同规则 |
|-----------|-----------|---------|
| 结构化审视工作质量和思维过程 | `criticism-self-criticism` | 人维度的反思——判断过程有没有问题？ |
| 在不确知环境中自动调整到最优 | `adaptive-robust-strategy` | 系统维度的自校正——方案本身有没有内置的调节机制？ |

**偏差层次→回退对照表：**

| 偏差层次 | 回退到 |
|---------|-------|
| 执行偏差（计划没错，执行不到位） | 回到「执行」环，调整资源或节奏 |
| 求解偏差（最优解在模型内对但在现实中错） | 回到「求解」环，检查约束是否遗漏 |
| 建模偏差（模型本身不能反映现实） | 回到「建模」环，修正系统描述 |
| 理解偏差（本质或主要矛盾判断错误） | 回到「理解」环，重新做第一性分析 |
| 感知偏差（根本不知道自己在面对什么） | 回到「感知」环，重新调查 |

## 认知环状态机

### 状态声明格式

```
📍 当前环节：[感知/理解/建模/求解/执行/校验/修正]
📤 上一环节输出：[简述]
✅ 进入下一环的条件：[已满足 / 未满足，缺少什么]
```

### 跨环宣告示例

```
感知完成。已知：系统有三个子系统，强耦合区域在 B-C 接口。
未知：子系统 A 的内部参数无法直接获取。
进入理解环。
```

### 环节跳跃规则

- 不允许跳跃过未走完的环节，除非深度判据已明确该环节不需要展开
- 在一个环节中遇到阻塞且无法在本环节内解决，回退到上一环节
- 修正环结束后，根据偏差层次决定回退到哪个环节
- 连续两轮经过同一环节而没有获得新信息，宣告认知环停滞，不再循环

### 深度等级→状态机行为

| 深度 | 状态机行为 |
|------|----------|
| 0 | 不启用状态机，直接行动 |
| 1 | 仅走感知环，完成即停 |
| 2 | 感知→理解→执行，每环宣告一次 |
| 3 | 全环顺序推进，每环宣告 + 出口条件检查 |
| 4 | 全环迭代，每次回环修正时重新宣告位置 |
