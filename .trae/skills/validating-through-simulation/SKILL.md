# 通过仿真验证

## 描述

未经验证的模型只是理论推测。本技能在模型建立后、实际部署前激活，通过结构化的多场景仿真实验检验模型正确性，用定量偏差分析指导修正，形成"假说→实验→修正→再实验"闭环。

## 使用场景

### 何时使用

- 模型已建立，需要验证其在多种场景下的行为
- 需要评估边缘情况和异常输入下的系统表现
- 需要在决策前对优化方案进行压力测试
- 模型经过修正后需要回归验证

### 何时不使用

- 还没有可测试的模型（应先完成定量建模）
- 测试范围极小，可直接在生产环境中低成本验证
- 系统形式化程度不够，无法构建有意义的仿真
- 模型可靠性已通过大量历史数据充分验证

## 输入输出

### Input

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| model | object | 是 | 待验证的模型（含版本号） |
| simulation_goal | string | 是 | 本轮仿真要验证的假设或回答的问题 |
| scenarios | array | 否 | 预定义仿真场景列表；未提供时按模板生成 |
| tolerance | object | 否 | 各指标的允许误差范围，默认正常±5%/边界±10%/压力±20% |
| environment_config | object | 否 | 仿真环境配置；未提供时需在准备阶段确定 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| simulation_plan | object | 仿真目标 + 场景列表 + 每场景预期结果和允许误差 |
| environment_checklist | object | 环境配置清单（模型版本、参数、随机种子、依赖版本） |
| results_summary | array | 逐场景结果摘要（通过/未通过、关键偏差、根因分析） |
| sensitivity_analysis | object | 关键参数变动对结果的影响 |
| validation_status | string | 验证结论：通过/需修正/需补充场景/发现关键未知 |
| traceability_records | array | 可追溯性记录（运行ID、时间戳、输入快照、环境信息） |

## 指令

1. **制定仿真计划**：明确仿真目标（要验证什么假设、回答什么问题）；按场景模板列出仿真场景——S1 正常（标准运行条件，基准性能）、S2 边界（极端但合法的输入，鲁棒性）、S3 异常（非法或意外输入，安全性）、S4 压力（超出设计限制，降级行为）、S5 综合（多因素同时变化，涌现行为）；为每场景设定预期结果和允许误差。
2. **准备仿真环境**：锁定模型版本（版本号或 commit hash）；固定参数文件和随机种子；确认计算环境一致（硬件、编译器、依赖库版本）；锁定输入数据版本和仿真脚本版本。
3. **按优先级执行仿真**：先执行 S1 正常场景验证基本正确性——未通过则先修正模型再继续；再执行 S2 边界场景验证鲁棒性；然后执行 S3 异常场景验证安全性；最后执行 S4/S5 压力测试和综合场景。
4. **分析结果**：对每场景进行定量对比（仿真值 vs 预期值，计算偏差和偏差率）；记录定性观察（行为模式、涌现行为、时间演变轨迹）；对超出允许范围的偏差进行异常归因——判定属于模型假设不合理、参数估计不准、数值方法误差、遗漏关键机制或预期值本身有误。
5. **决定模型修正方向**：若偏差在允许范围内且场景覆盖充分，进入工程编排；若偏差超出允许范围，修正模型后重跑全部场景；若场景覆盖不充分，先补充场景再判断是否需修正模型；若发现关键未知，返回问题域调研。
6. **生成验证记录**：记录每次仿真运行的时间戳、输入参数快照、关键输出值、运行环境信息；确保给定相同输入可复现相同输出；模型修正后对所有已通过场景执行回归测试。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 正常场景即未通过，无法继续后续场景 | 立即停止后续场景执行，返回定量建模修正模型基本逻辑，修正后从 S1 重新开始 |
| 仿真结果不可复现（相同输入不同输出） | 检查随机种子是否固定、是否存在并发竞争或浮点精度问题，修复后重新执行 |
| 场景覆盖不足但无法构造更多场景 | 标记为"覆盖不足"，在验证报告中明确未覆盖的风险域，建议在工程编排阶段增加监控点 |
| 偏差归因无法确定（多个可能原因交织） | 启用 investigation-first 对偏差进行隔离实验，逐因素控制变量定位根因 |

## 示例

### 示例一：调度算法模型验证

**仿真目标**：验证调度算法在正常负载和峰值负载下的任务完成率。

**场景设计**：S1 正常负载 100 任务/小时，预期完成率≥95%，允许误差±5%；S2 峰值负载 500 任务/小时，预期完成率≥80%，允许误差±10%；S3 异常输入（无效任务），预期优雅降级不崩溃。

**执行结果**：S1 通过（完成率 96.2%）；S2 未通过（完成率仅 62%，偏差 18%）；S3 通过。

**异常归因**：S2 偏差根因为算法在队列满时采用 O(n²) 重调度，导致处理延迟非线性增长。

**修正方向**：修正模型——将重调度策略改为 O(n log n) 优先队列；修正后回归测试 S1 和 S2。

### 示例二：供应链模型压力测试

**仿真目标**：验证供应链模型在供应商突发断供时的恢复能力。

**场景设计**：S4 压力场景——主供应商断供，预期 7 天内恢复至 70% 产能，允许误差±20%。

**执行结果**：恢复需 14 天，偏差 100%，严重超标。

**归因**：模型中备选供应商切换时间假设为 1 天，实际需 5 天审批流程。

**修正方向**：修正模型参数——备选供应商切换时间从 1 天调整为 5 天；补充 S5 综合场景（主供应商断供+需求激增同时发生）。
