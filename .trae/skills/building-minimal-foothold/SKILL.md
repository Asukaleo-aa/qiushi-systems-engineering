# 星火燎原

## 描述

从小处着手不等于没有前途。关键是建立稳固的"根据地"——一个可站稳、可复用、可扩展的基础——而不是四处游击、散乱行动。在正确的条件下，星星之火完全可以发展为燎原之势。

## 使用场景

### 何时使用

- 从零开始一个新项目/新任务
- 资源极度有限，看起来不可能完成大目标
- 面对宏大目标不知从何下手
- 正在做很多零散的事但没有积累
- 需要用最小的投入验证一个想法的可行性

### 何时不使用

- 项目已有稳固基础，处于扩展阶段而非从零开始
- 用户已明确切入点，只需执行
- 任务是对现有系统的修改——现有代码就是"根据地"
- 已有充分资源，不需要谨慎起步策略

## 输入输出

### Input

| 字段 | 类型 | 说明 |
|------|------|------|
| goal | string | 最终要达成的目标 |
| current_resources | string | 当前拥有的资源描述 |
| constraints | list[string] | 已知的约束条件 |

### Output

| 字段 | 类型 | 说明 |
|------|------|------|
| condition_assessment | {demand_exists, foundation, obstacles, verdict} | 客观条件评估 |
| foothold | {choice, advantage, stability_evidence, expansion_path} | 选定的根据地 |
| roadmap | list[{step, success_signal}] | 分步发展路线（不超过3步） |

## 指令

1. **评估客观条件**：判断需求是否真实存在、现有可依托的基础是什么、主要障碍有哪些。条件不具备时先调用 investigation-first 创造条件，不急于行动。
2. **选定根据地**：选择你最有优势的切入点，确保该点能站稳（不只是做一次就扔掉）且能扩展（从这里可向外延伸）。具体场景映射：新项目→最核心最小可用功能模块；新领域→一个子领域的深入理解；新能力→一个基础技能作为起点。
3. **拒绝流寇主义**：每次扩展前检查——上一个切入点是否已验证稳固？这次扩展是否从上一个根据地自然延伸？新增内容是否可复用？任一不满足则不得扩展。
4. **扎下根来**：在一个点上深入扎根，每一步成果都为下一步打基础，建立可复用的基础设施和知识，保持明确的扩展路径。
5. **积小胜为大胜**：每次扩展一小步，确保站稳后再向外延伸，保持耐心，记录每次扩展的经验。
6. **拒绝两种错误**：悲观主义（"资源太少做不了什么"）——条件具备时小力量完全可以发展壮大；冒险主义（"到处出击先干再说"）——没有根据地的行动是盲目的，先建基础再发展。

## 失败策略

| 失败场景 | 处理方式 |
|---------|---------|
| 客观条件评估后发现需求不真实 | 暂停行动，调用 investigation-first 重新调研，或调整方向寻找真实需求 |
| 选定的根据地无法站稳（验证失败） | 回退到条件评估步骤，重新选择更有优势的切入点，不强行推进 |
| 根据地还没站稳就面临扩展压力 | 明确拒绝扩展请求，输出当前根据地的稳固性检查结果，说明未稳固扩展的风险 |
| 多个候选切入点各有优劣无法抉择 | 选择依赖关系最少的那个（最孤立、最易站稳），或调用 contradiction-analysis 分析主次 |

## 示例

### 示例一：从零启动新项目

目标：构建一个全栈任务管理应用。资源：1人、2周。

- 条件评估：需求真实（团队确实需要），现有基础为零，主要障碍是时间有限。结论：具备发展条件。
- 根据地：命令行版任务增删查改，选择理由——最小可用、可验证核心逻辑、后续可扩展为 Web 版。
- 路线：第1步 CLI 增删查改→成功标志：可正常添加和列出任务；第2步 持久化存储→成功标志：重启后数据不丢失；第3步 Web 界面→成功标志：浏览器可操作任务。

### 示例二：进入陌生技术领域

目标：掌握机器学习。资源：业余时间、无相关背景。

- 条件评估：需求真实（项目需要），现有基础仅有编程能力，主要障碍是知识缺口。结论：需先创造条件。
- 根据地：用 scikit-learn 完成一个分类任务，选择理由——最小可运行示例、建立直觉、可扩展到深度学习。
- 路线：第1步 跑通一个分类示例→成功标志：理解输入输出含义；第2步 调参并理解原理→成功标志：能解释模型行为；第3步 应用到实际数据→成功标志：产出可用结果。
