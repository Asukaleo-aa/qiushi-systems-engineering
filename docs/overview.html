项目概述

实事求是 + 系统工程 + 工程控制论 → AI Agent 方法论工具链

qiushi-systems-engineering 是一套为 AI Agent 设计的方法论 skill 集合。28 个 skill 覆盖 agent 认知闭环的七个环节，使 agent 在面对复杂任务时能够：摸清现状再判断、抓住本质和主要矛盾、建立系统模型、在约束下求最优解、有序推进不散乱、验证偏差不盲信、根据反馈自校正。

核心洞察：AI 模型能力的增强掩盖了方法论的匮乏。Agent 在复杂任务中反复试错，缺乏系统性的判断框架。本体系提供的不是"告诉 Agent 做什么"，而是"教 Agent 如何思考"——一套从感知到修正的结构化认知流水线。

体系的组织逻辑不是「毛泽东的方法 vs 钱学森的方法」，而是按 agent 的实际认知流程组织。在每个认知环节下，毛泽东的哲学方法论和钱学森的工程方法论协同工作、互补而非互斥。

哲学根基

毛泽东哲学方法论
《实践论》《矛盾论》提供认识论根基：实事求是、调查研究、矛盾分析、实践检验。教 Agent 如何接近真理——不是从教条出发，而是从事实出发。
《实践论》1937 · 《矛盾论》1937 · 《改造我们的学习》1941
钱学森系统工程方法论
《工程控制论》《论系统工程》提供工程方法：系统思维、最优控制、状态估计、反馈修正。教 Agent 如何系统地解决问题——不靠枚举，靠结构化的最优性条件。
《工程控制论》1954/1980 · 《论系统工程》1982

两者汇聚为一条总原则——实事求是：先看事实，再下判断。

箭头方向：感知→理解→建模→求解→执行→校验→修正→（回环至感知）

状态机机制：每个环节有明确的出口条件。偏差发生后，根据偏差层次自动回退到对应环节（执行偏差→执行环、建模偏差→建模环、理解偏差→理解环、感知偏差→重新调查）。连续两轮经过同一环节无新信息则宣告停滞。

设计原则

01 · 认知环组织
按 agent 的实际认知流程组织技能，而非按思想来源分类。毛泽东方法和钱学森方法在同一认知环节下互补。
02 · 双源协同
毛泽东哲学和钱学森工程不是两个流派，而是同一认知目标下的两套互补方法。每个环节都可调用双方工具。
03 · 分层不割裂
每个技能独立可用，也可以完整串联。不强制走全环——简单任务只走感知+执行，复杂任务全环展开。
04 · 原著为锚
每个 skill 标明在原著中的具体出处（章节号或文章名），确保方法论根基可追溯、可验证。
05 · 操作化输出
每个 skill 包含明确的步骤、触发条件、不适用场景和常见错误。不是哲学散文，是可执行的规程。
06 · 承认不知道
不确定时返回调查，不用猜测填补。宁可说"这个需要查"也不给出没有依据的结论。
07 · 反模板化
在套用任何已有方案前，先列出当前情境与过往案例的实质性差异。直接复用旧方案而不检查前提是否成立，是框架禁止的。

v3.1 · 系统层次意识

防止"把产品当物理系统优化"类层次错位

v3.1 的核心升级：在对系统建模之前，先锚定系统在哪个层次上被讨论。同一对象在不同层次上，目标函数不同、约束集不同、最优解不同。

物理系统
由自然定律封闭。目标来自内部物理量（效率、比冲、功率密度）。约束为守恒定律和材料本征极限。
工程系统
物理系统 + 制造/装配/维护可行性。目标为性能-可制造性权衡。约束增加工艺极限、供应链、公差。
产品系统
工程系统 + 使用场景 + 生命周期。目标由使用场景赋予（$/kg to orbit）。约束增加用户需求和经济可行性。
社会-技术系统
产品系统 + 组织/市场/制度。目标为长期存续与演化能力。约束增加产业生态、法规、资本效率。

升级涉及 7 个关键节点：网关判据（arming-thought）、系统思维框架第零步（systems-thinking-framework）、约束分类（first-principles-analysis）、边界六维检查（system-boundary-structuring）、目标函数层次校验（constrained-optimization）、输入层次标注（meta-synthesis-engine）、场景层根因（feedback-and-revision-loop）。

技能体系 · 28 Skills

按七环认知闭环组织 · 每环标注毛/钱来源

PERCEIVE 感知 · 我面对的是什么？

摸清现状、收集事实、从噪声中推断隐藏状态。先调查再发言。

investigation-first · 调查研究
先调查再发言，列提纲逐项摸清。定义观测框架：看什么、问谁、怎么验证。信息不充分时严禁下判断。
毛泽东《反对本本主义》
信息缺口、证据不足、领域陌生、需要先摸清现状
problem-domain-investigation · 问题域调研
进入陌生领域前的系统性调研。建立知识地图，评估信息来源可靠性，区分已知/未知/推测。
钱学森《论系统工程》第25章
领域知识不足、信息缺口明显、需区分已知未知推测
mass-line · 群众路线
收集→集中→返回→检验。整合多方意见，从分散的个体反馈中凝聚为系统判断。
毛泽东《关于领导方法的若干问题》
多方意见、stakeholder 反馈、需要收集-综合-验证循环
state-estimation · 状态估计
从噪声、矛盾、残缺数据中递推推断隐藏状态。维纳滤波、卡尔曼滤波——在信息残缺条件下推测真实情况。
钱学森、宋健《工程控制论》第14-15章
数据噪声大、信息源矛盾、需在线递推更新认知

UNDERSTAND 理解 · 本质是什么？主要矛盾在哪？

剥离假设到不可再分的基本元素，追溯系统演化路径，识别核心张力。

first-principles-analysis · 第一性原理分析
剥离所有假设到不可再分的基本元素，再从基本元素向上重建理解。区分物理约束、场景约束、经济约束、工程惯例和历史遗留。
亚里士多德/笛卡尔/钱学森《工程控制论》第1章
全新问题、框架无法套用、假设堆积需打散重建
historical-evolution-analysis · 历史演化分析
追溯因果链→识别路径依赖→分离偶然遗留与结构必然。理解系统"为什么是现在这个样子"。
钱学森《论系统工程》《工程控制论》
理解旧系统、重构前需理解演化路径、区分历史惯性与结构必要
contradiction-analysis · 矛盾分析
识别矛盾→判定主要矛盾→区分对抗性/非对抗性→确定主攻方向。主要矛盾 = 约束瓶颈 = 优先攻克的目标。
毛泽东《矛盾论》
问题复杂、多因素冲突、优先级不清、根因不明

MODEL 建模 · 怎么描述这个系统？

建立系统世界观，定义边界、组分、层次、涌现和反馈。从定性认识跃迁到可计算模型。

systems-thinking-framework · 系统思维框架
第零步确定系统层次（物理/工程/产品/社会-技术），然后六步建立系统世界观：目的→边界→组分→层次→涌现→反馈。
钱学森《论系统工程》
多子系统、目标模糊、需整体把握再分解
system-boundary-structuring · 系统边界结构
层次分解→组分规格→接口契约→六维边界检查（物理/逻辑/时间/组织/知识/场景）。
钱学森《论系统工程》第37章
组分超过5个、多层级嵌套、接口关系复杂
quantitative-modeling-workflow · 定量建模
从经验性假设和定性描述出发，识别变量→选择建模方法→构建可计算模型。定性假设→定量模型。
钱学森《论系统工程》第1、2、8章
需将系统转化为数学模型、需优化或预测
meta-synthesis-engine · 综合集成引擎
定性判断 + 定量结果 + 经验 → 统一结论。标注每个输入来源的系统层次，处理跨层次冲突（层面错位≠真分歧）。
钱学森《论系统工程》增订版
多来源矛盾、定性定量需统一、需升华整体判断

SOLVE 求解 · 在约束下，最优方案是什么？

目标清晰后，在约束空间内寻找最优策略。受控系统的最优控制通常触碰约束边界而非停留在内部。

constrained-optimization · 约束优化
先校验目标函数层次（产品系统不能用物理层指标），再用庞特里亚金极大值原理或贝尔曼动态规划求解。最优解通常触碰约束边界。
钱学森、宋健《工程控制论》第8-9章
需选最优方案、资源有限、在约束下最大化指标
hierarchical-decomposition-coordination · 递阶分解协调
将大系统分解为自治子系统，加协调层防止局部优化导致全局恶化。分散化控制——子系统自治，协调器只处理耦合。
钱学森、宋健《工程控制论》第21章
多子系统互扰、局部优化恶化全局、需分而治之
duality-complementarity-analysis · 对偶互补分析
正向推导不通→构造对偶问题→在对偶空间求解→翻译回原空间。状态↔协态、能控↔能观测、原系统↔伴随系统。
钱学森、宋健《工程控制论》第4/8/13章
正向路径受阻、问题有对称结构、"正难则反"
perturbation-progressive-method · 摄动渐进
先解标称（简化）问题→将差异视为小摄动→一阶校正→评估摄动假设何时失效→逐步加回复杂因素。
钱学森、宋健《工程控制论》第13章
精确求解不可行、需从简单情形逐步逼近
multi-representation-equivalence-transform · 多表述等价变换
在一种数学工具下卡住时，切换到等价表述。时域↔频域↔状态空间↔几何↔希尔伯特空间。选择使核心操作最简单的表述。
钱学森、宋健《工程控制论》第2章
同一方向反复无果、需切换数学视角

EXECUTE 执行 · 怎么落地？

将方案编排为有依赖关系的任务序列，确定主攻方向，集中兵力逐一歼灭。

engineering-orchestration · 工程编排
WBS→关键路径→质量门→风险矩阵，将方案编排为有依赖关系和里程碑的可执行任务序列。
钱学森《论系统工程》
需制定开发计划、任务过多需编排调度
concentrate-forces · 集中兵力
多任务中确定唯一主攻方向，停止分散用力。在每个阶段只有一个中心目标，其他都绕着它转。
毛泽东十大军事原则
优先级过多、资源紧张、推进分散
spark-prairie-fire · 星火燎原
从零起步，找最小可行切入点建立根据地。先验证核心假设，再从验证过的核心向外扩展。
毛泽东《星星之火，可以燎原》
资源极少、需 bootstrap/MVP/pilot
protracted-strategy · 持久战略
长期复杂任务分三阶段：防御（站稳脚跟）→相持（积累力量）→反攻（实现突破）。不追求速胜，保持战略耐心。
毛泽东《论持久战》
目标长期、短期无法速胜但不可放弃
overall-planning · 统筹兼顾
多个目标相互制约时做动态平衡。不是单目标最优，而是十个指头弹钢琴——各有轻重缓急但都在动。
毛泽东《论十大关系》
多目标冲突、优化一项伤害另一项

VERIFY 校验 · 做得对吗？

结果与预期比对，偏差量化，根因追溯。真理的标准只能是社会实践。

practice-cognition · 实践认识论
实践→认识→再实践的螺旋上升。感性认识（积累材料）→理性认识（提炼规律）→实践验证→总结升华。全环节的方法论约束。
毛泽东《实践论》
方案需验证、假设需检验、需迭代升级认知
simulation-validation-cycle · 仿真验证
多场景仿真 + 边界条件测试 + 压力测试。在实际执行前通过仿真发现模型的薄弱环节和失效条件。
钱学森《论系统工程》第43章
模型已建立、需在多种场景下测试
feedback-and-revision-loop · 反馈修正循环
阶段偏差量化→五层根因追溯（哲学/理论/场景/方法/执行）→三级修正（短期止血/中期治本/长期免疫）→认知更新。
钱学森《论系统工程》
阶段完成需审查、执行结果与预期不符

CORRECT 修正 · 怎么改？

基于偏差的层次决定回退到认知环的哪个环节。修正不是惩罚，是方法论自愈。

criticism-self-criticism · 批评与自我批评
结构化审视工作质量和思维过程。具体行为归因→根本原因→改进建议。目的不是追责而是"治病救人"。人维度的反思。
毛泽东《整顿党的作风》《论联合政府》
工作完成需审视、收到反馈、反复出错需纠偏
adaptive-robust-strategy · 自适应策略
自寻优→自行镇定→模型参考自适应。在环境参数不可预知变化时，系统自动保持最优。系统维度的自校正。
钱学森、宋健《工程控制论》第16/18章
环境变化不可预知、运行中性能下降需自校准

META 元能力 · 总原则与工作流

arming-thought · 武装思想
实事求是总原则 + 认知环路由器。在每次顶层对话开始时自动加载。包含六级深度判据（确定展开哪些认知环）和系统层次定位（物理/工程/产品/社会-技术）。
毛泽东 + 钱学森 方法论的统一入口
每次新的顶层对话开始时自动调用
workflows · 标准化工作流
三条标准化跨环节工作流：① 新项目启动（感知→理解→建模→求解→执行）② 复杂问题攻坚（感知→理解→求解→校验→修正循环）③ 方案迭代优化（执行→校验→修正→建模→求解循环）。
技能体系的编排层
任务需跨环节串联多个 skill

偏差回退机制

当校验环发现偏差后，根据偏差的层次决定回退到认知环的哪个位置：

执行偏差
计划没错，执行不到位 → 回到执行环，调整资源或节奏
求解偏差
最优解在模型内对但在现实中错 → 回到求解环，检查约束是否遗漏
建模偏差
模型本身不能反映现实 → 回到建模环，修正系统描述
场景偏差
错误源于在错误的系统层次上定义问题 → 回到系统思维框架第零步，重新锚定层次
理解偏差
本质或主要矛盾判断错误 → 回到理解环，重新做第一性分析
感知偏差
根本不知道自己在面对什么 → 回到感知环，重新调查

自指性：方法论的终极检验

Self-Reference

qiushi 框架最独特的特征：它可以检验和升级自己。当框架本身产出错误时，自身的校验-修正链路能够定位根因，并推动框架的结构性迭代。

典型案例：在讨论液体火箭发动机优化时，框架按纯物理系统给出了最大化比冲的解。用户指出"发动机是一件产品"——这暴露了框架缺少"系统层次意识"的结构性缺陷。随后的复盘用框架自身的 criticism-self-criticism 定位了根因，用 feedback-and-revision-loop 追溯了七个关键节点的缺失，最终推动了 v3.1 的升级。

实践→认识→再实践→再认识——这个螺旋不仅适用于框架分析的对象，也适用于框架自身。
