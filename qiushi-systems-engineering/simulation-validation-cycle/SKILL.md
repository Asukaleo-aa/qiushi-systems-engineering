# 仿真验证循环

> "孙武是我国古代一位伟大的军事哲学家。他在写作不朽的军事哲学著作《孙子兵法》过程中，事实上是在自己的脑子里按不同的战斗条件推演了矛盾斗争的整个过程。他是军事史上第一个运用模拟作战的思想方法研究战争的伟大人物。"
> —— 王寿云《作战模拟方法学在中国》

> "作战理论研究的一般过程是：从现实提取研究素材...进行理论分析...在理论分析基础上提出新的作战思想..."
> —— 王寿云《以理论为依据的需求体制》

## 核心原则

**不动手验证的模型只是理论推测。仿真验证是"实践是检验真理的唯一标准"在系统工程中的具体实现。必须用可重复的实验来检验模型的正确性，用定量的误差分析来指导模型的修正。**

本 skill 建立了结构化的"假说→实验→修正→再实验"闭环，是 qiushi `practice-cognition`（实践认识论）在工程技术层的直接实现——仿真就是系统工程的"实践"，偏差分析就是系统工程的"再认识"。

---

## 认识论基础（来自 qiushi 的 `practice-cognition`）

在进入本 skill 的技术流程之前，应先通过 `practice-cognition` 明确当前处于实践认识循环的哪个阶段：

- **感性认识阶段** → 对应本 skill 的"执行仿真，积累数据"（第一步到第三步）
- **理性认识形成阶段** → 对应本 skill 的"结果分析，形成规律"（第四步）
- **实践验证阶段** → 对应本 skill 的"回归测试，确认修正"（第五步回归）
- **总结升华阶段** → 对应本 skill 的"生成验证记录，提炼教训"（第六步）

`practice-cognition` 提供"为什么这样做"的认识论理由，本 skill 提供"具体怎么做"的工程技术规程。

---

## 不适用场景

以下情况**不需要**调用本 skill：
- 还没有可测试的模型（应先调用 quantitative-modeling-workflow）
- 测试范围极小，可以直接在生产环境中低成本验证
- 系统的形式化程度不够，无法构建有意义的仿真
- 模型的可靠性已经通过大量历史数据验证

## 方法论流程

### 第一步：制定仿真计划

> "从现实提取研究素材，如军事历史、战例经验、演习资料等；然后进行理论分析" —— 仿真必须有目的，有方案

**仿真计划模板：**

```
仿真目标：[要验证什么假设？回答什么问题？]

仿真场景列表：
| 场景编号 | 场景描述 | 为什么关键 | 预期结果 | 允许误差 |
|---------|---------|-----------|---------|---------|
| S1 正常 | 标准运行条件 | 基准性能 | ... | ±5% |
| S2 边界 | 极端但合法的输入 | 鲁棒性 | ... | ±10% |
| S3 异常 | 非法或意外输入 | 安全性 | ... | 定性检查 |
| S4 压力 | 超出设计限制 | 降级行为 | ... | ±20% |
| S5 综合 | 多因素同时变化 | 涌现行为 | ... | 定性观察 |

仿真次数：每场景至少运行 N 次（单体仿真）或 M 个随机种子（随机仿真）
```

### 第二步：准备仿真环境

**确保仿真的可复现性：**

```
环境清单：
- [ ] 模型版本锁定（版本号或commit hash）
- [ ] 参数文件固定（所有参数值已记录）
- [ ] 随机种子固定（如果需要可控随机性）
- [ ] 计算环境一致（硬件、编译器版本、依赖库版本）
- [ ] 输入数据版本固定
- [ ] 仿真脚本版本控制
```

### 第三步：执行仿真

**按优先级顺序执行场景：**

1. 先跑 S1（正常场景）—— 验证基本正确性
2. 再跑 S2（边界场景）—— 验证鲁棒性
3. 然后 S3（异常场景）—— 验证安全性
4. 最后 S4/S5 —— 压力测试和综合场景

**如果基本场景未通过，先修正模型再继续。**

### 第四步：结果分析

> 钱学森强调系统工程是"实干的"，"要取得实际效果"

**对每个场景进行结构化分析：**

```
场景Sx结果分析：

定量对比：
| 指标 | 仿真值 | 预期值 | 偏差 | 偏差率 | 是否通过 |
|------|--------|--------|------|--------|---------|
| KPI1 | xxx    | yyy    | aaa  | ± bb%  | ✓/✗     |

定性观察：
- 行为模式是否符合预期？ [是/否，说明]
- 是否出现了意外的涌现行为？ [是/否，说明]
- 时间演变轨迹是否合理？ [是/否，说明]

异常归因：
- 如果偏差超出允许范围，根本原因是：
  [ ] 模型假设不合理
  [ ] 参数估计不准
  [ ] 数值方法误差
  [ ] 遗漏了关键物理/逻辑机制
  [ ] 预期值本身有误
```

### 第五步：模型修正决策

**基于仿真结果决定下一步：**

```
决策矩阵：
                    偏差在允许范围内        偏差超出允许范围
                    ──────────────        ──────────────
场景覆盖充分      → 进入工程编排           → 修正模型后重跑

场景覆盖不充分    → 补充仿真场景           → 先补充场景
                                               再判断是否需要修正模型
```

### 第六步：生成验证记录

**确保仿真的可追溯性：**

每个仿真运行必须记录：
- 运行时间戳
- 输入参数快照
- 关键输出值
- 运行环境信息
- 执行者/触发者

## 质量检查表

- [ ] 每个仿真场景有明确的通过/未通过标准？
- [ ] 仿真可以完全复现（给定相同输入，得到相同输出）？
- [ ] 覆盖了正常、边界、异常、压力四类场景？
- [ ] 偏差超过允许范围时，有根因分析？
- [ ] 模型修正后，所有已通过的场景都重新跑过（回归测试）？
- [ ] 仿真结果存档，可供后续追溯？

## 与其他 skill 的关系

### 系统工程链内

- **上游**：接收 `quantitative-modeling-workflow` 的模型
- **下游**：验证通过的模型传递给 `engineering-orchestration` 进入执行
- **循环**：验证未通过时触发 `feedback-and-revision-loop`，决定返回建模或调研
- **整合**：验证结论传递给 `meta-synthesis-engine` 作为综合判断的一部分

### 与 qiushi 思想方法层深度耦合

| 本 skill 的步骤 | 联动 qiushi skill | 联动方式 |
|---------------|------------------|---------|
| 仿真计划 | `practice-cognition` | 宣告当前实践认识循环阶段，明确"本轮要验证什么" |
| 结果分析 | `contradiction-analysis` | 偏差与预期的矛盾→矛盾性质判定 |
| 异常归因 | `investigation-first` | 根因不明时，回到实地调查 |
| 模型修正决策 | `practice-cognition` | 修正后回到"再实践"循环 |
| 验证记录 | `criticism-self-criticism` | 仿真通过/未通过→工作质量审视 |

**核心闭环：**
```
practice-cognition （宣告阶段）
    ↓
simulation-validation-cycle （执行仿真）
    ↓ （如果通过）
practice-cognition （"认识基本正确"，进入下一轮）
    ↓ （如果未通过）
feedback-and-revision-loop （系统偏差根因分析）
    ↓
quantitative-modeling-workflow （修正模型）
    ↓
simulation-validation-cycle （回归验证）
    ↓
criticism-self-criticism （工作审视：为什么第一次没通过？）
```
